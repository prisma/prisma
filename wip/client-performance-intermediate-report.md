# Prisma Client performance intermediate report

Date: 2026-06-24.

This report summarizes the current magnitude of the performance and memory gains from the ongoing Prisma Client performance branch. The numbers are from the persistent journal and focused probes in this branch; many rows are microbenchmarks over fake adapters or Miniflare/workerd harnesses, so they should be read as directional product-path evidence rather than final customer benchmarks.

## Executive Snapshot

Current as of 2026-06-24, after the latest Prisma docs commits, engines `29000399a8a`, the strict flat string `findFirst` generated prepared helper slice, and the post-harness-restart generator verification refresh.

The largest proven gain is still avoiding query compilation on hot requests. Local `ClientEngine` timing moved `findUnique` from cache-disabled to warmed cache at `752.61 -> 16.49 us/op` and the nested blog-page shape at `3122.43 -> 37.16 us/op`, roughly `46x` and `84x` faster than recompiling every request.

The important steady-state generated-client nested rows have roughly doubled, but are not uniformly at the 3x target yet. The by-author blog-feed path moved generated default / request-precomputed / exact-helper from `21.97 / 20.35 / 16.58 us/op` to `10.17 / 10.62 / 9.31 us/op`; direct plan execution moved `13.70 -> 6.95 us/op`. The newest post-restart smoke kept the same ordering at generated default / generated prepared / exact-helper `11.09 / 8.70 / 9.89 us/op`, with direct plan at `7.26 us/op`.

The newest accepted generated cache-hit slice adds strict flat string `findFirst` / `findFirstOrThrow` prepared helpers behind `internalExactDescriptorHelpers`. A corrected 300k Node `findFirst users` run measured default / request-precomputed / exact-helper / generated prepared at `2.49 / 2.42 / 2.37 / 1.39 us/op`, with prepared direct/request-surface lower bounds at `1.09 / 1.40`. The matching 20k Workerd request-loop run measured default / exact-helper / prepared direct / generated prepared at `2.95 / 1.90 / 1.15 / 1.45 us/op`.

Plan-cache memory is materially lower. The blog-page node warm retained heap moved from about `18.75 MiB` before the compact/interner work to a current-band `4.48 MiB`, about `76%` lower. Serialized retained plan JSON for the same probe moved from about `9.45 MiB` to `3.33 MiB`, about `65%` lower.

The biggest contributors are: warmed query-plan caching; compact lockstep-only plan formats plus string/result sharing; raw-nested and final-owner execution; generated/prepared exact cache-hit helpers; edge plain delegates for extension-free Workerd clients; and Rust query-compiler semantic phase deletions for M:N `set` / disconnect, upsert final reads, nested-upsert shared connects, required nested `set`, FK-backed nested updates, and raw-nested cursor/distinct relation ops.

The latest accepted generated-client speed delta is the flat string `findFirst` prepared helper above. The latest accepted compile-path speed delta remains FK-backed required to-one nested update in engines `f23bb7406c7`, moving `update-set-nested-prisma#27650` full-compile allocations `1621 -> 1453` and Criterion about `153.20 -> 135.75 us`. Engines `c700d8ffb68` is runtime protocol groundwork: it emits a raw-nested final-owner schedule marker consumed by TS, but it is not the dedicated schedule executor that should move the headline feed timings.

The main remaining gap is not another shallow conversion like `serde_wasm_bindgen::from_value()`: that would still build Rust-owned `JsonBody` / map structures and would not remove JS cache-key stringification. The next plausible large wins are expanding generated prepared/exact cache-hit coverage to more flat hot shapes, a dedicated generated/static final-owner schedule executor on top of the new marker, a JS-owned cache-hit/query-input architecture that avoids Wasm on hits, or fresh profile-backed Rust graph/translation ownership work.

The latest post-restart M:N connect work does not change the headline. Replacing nested connect filter dedupe with a one-pass `Vec::contains()` loop saved only two full-compile allocations on `update-m2m-connect` (`1442 -> 1440`) and adjacent connect-heavy rows, while focused Criterion was mixed and softened `create-nested-connectOrCreate-*` rows, so it was reverted. The accepted follow-up is prerequisite SQL-builder work: engines `d59deb11278` renders Postgres `INSERT ... SELECT ... RETURNING ...`, and engines `29000399a8a` lets Quaint put `INSERT ... RETURNING` inside a CTE. A direct `ConnectRecordsByFilter` CTE phase-owner prototype then passed checks and query snapshots, but worsened full-compile allocations on `update-m2m-connect` (`1442 -> 1509`) and `nested-upsert-nested-only` (`2555 -> 2621`), so it was reverted too. The real M:N connect lead remains valid only if the next version changes the allocation profile or proves a larger runtime/DB-roundtrip win.

## Current Rollup

The largest proven end-to-end win remains the warmed query-plan cache path itself: the local timing probe measured `findUnique` cache-disabled vs warmed-cache at `752.61 -> 16.49 us/op` and the nested blog-page shape at `3122.43 -> 37.16 us/op`, roughly `46x` and `84x` faster than recompiling every request.

On the hot cached generated nested feed path, later product-path work roughly halved runtime again. The journaled by-author feed A/B moved generated default / request-precomputed / exact-helper from `21.97 / 20.35 / 16.58 us/op` to a later confirmed `10.17 / 10.62 / 9.31 us/op`; direct plan moved `13.70 -> 6.95 us/op`. That is about `2.0-2.2x` faster on the warmed nested feed path.

Retained plan-cache memory is materially better as well. The `blog page / node default warm` memory probe moved from about `18.75 MiB` retained heap before string interning to a later current-band `4.48 MiB`, about `76%` lower. Serialized retained plan JSON for the same named probe moved from about `9.45 MiB` to `3.33 MiB`, about `65%` lower.

The biggest contributors are: warmed query-plan caching; compact plan shapes plus string interning; raw-nested/final-owner execution; generated prepared and exact descriptor helpers; semantic Rust graph removals for `set`, upsert, update, connect-or-create, nested-upsert, FK-backed nested updates, and narrow M:N relation-table operations; raw-nested relation-op emission for cursor/distinct pagination including M:N mapped operation-field remapping and one-to-many parent grouping; and parser/translation allocation cleanups. ## What Did Not Pay Off

Several plausible ideas were tested and rejected:

- `serde_wasm_bindgen` / one-pass JSON deserialization as a standalone path is unlikely to be the main win. The native allocation profile shows JSON parse/adaptation is smaller than `graph_build` and `translate_ir`, and cache hits still need JS-side cache-key/shape work anyway. A 2026-06-11 code scout confirmed the current request path still has no active `compileFromValue` entrypoint and that a `serde_wasm_bindgen::from_value()` replacement would still build owned `JsonBody` / `IndexMap<String, serde_json::Value>` / `Operation` / `ArgumentValue` trees before graph build.
- A direct `compileFromValue(JsValue)` Wasm entrypoint is also too shallow if it still builds today's owned Rust protocol maps. The 2026-06-10 release-Wasm spike measured current string compile misses versus JS-value compile misses at 404.42 vs 422.08 us/op for `findUnique`, 299.61 vs 300.55 for `findMany`, and 2306.57 vs 2348.81 for blog-page, so it was reverted.
- Cache-key-only work is low ceiling on Workerd: pure cache-key/lookup is about 0.4 us/op for `findUnique` and 1.5 us/op for blog-page, far below the full generated-client cost.
- Generic descriptor matchers, generic raw-nested flat schedules, and instruction-array executors that still allocate generic result/record/scope/slot arrays do not beat the current paths enough to productize.
- Cache-handle-only nested work is too small by itself. A fresh 2026-06-10 300k nested blog-page run measured `cached request wrapper static shape` at 6.92 us/op versus direct plan at 6.55 us/op, while `cached request wrapper static descriptor` was 8.10 and lazy descriptor was 9.42. Removing only `QueryPlanCache.getSingle()` / response-wrapper glue can at most recover a few tenths of a microsecond; it does not address descriptor extraction or raw-nested result execution.
- A generated generic exact-helper emitter is not enough by itself. The 2026-06-09 prototype passed focused generator tests and avoided `eval` / `new Function`, but its extractor microbench was mixed against the kept runtime factory and the broader generated-client timing run was too noisy to justify a large generated-code diff.
- JS-only parameterization of pagination `take` is not viable. A 2026-06-11 probe confirmed DMMF marks root/relation `take` and `skip` as non-parameterizable, and the current Wasm compiler rejects manually inserted `Param<Int>` values for root `take`, nested `comments.take`, shared root+nested placeholders, and separate root/nested placeholders with `Placeholder<Int> is not a valid Int`. Arbitrary `take` support would need an engines-side pagination placeholder design, not only a client cache-key change.
- Reordering descriptor extraction to try all exact matchers before lazy fallback is not enough. A 2026-06-11 patch targeted alternating nested-shape overhead, but exact-helper blog-page worsened 8.71 / 9.03 -> 9.09 / 9.16 us/op for stable/alternating rows, and descriptor-bound static worsened 8.97 / 9.91 -> 9.59 / 9.62. The reverted control returned exact-helper rows to 8.68 / 8.98.
- Special-casing nested exact helpers to bind only canonical `%1` and return a fixed `{ '%1': value }` object literal is not enough. A 2026-06-11 close A/B over 100k generated blog-page exact-helper iterations measured the patched benchmark mirror at 8.97 / 9.32 us/op for stable/alternating rows, while the dynamic-placeholder close control measured 8.43 / 9.37. This was reverted.
- Local raw-nested wrapper/leaf shortcuts are not enough on their own. Even the positive direct leaf shortcut stayed below the acceptance gate and left the generic `RawNestedReadResult` / attach-helper architecture in place.
- Direct raw-nested row-mapper loop unrolling is not enough. A 2026-06-10 spike unrolled numeric/direct mapping lengths 0-4 in `compileRawNestedRowMapper()`, but the patched 300k nested run regressed generated blog-page 11.75 -> 12.27 us/op, exact descriptor helper 10.21 -> 10.78, cached-wrapper static descriptor 8.10 -> 8.52, direct plan 6.55 -> 7.14, and only moved compact node 6.27 -> 6.33. This was reverted.

A 2026-06-11 patch returned straight-line closures for direct string-field final-owner mapping lengths 0-4, but patched direct / exact compact / generated nested rows measured 5.85 / 5.68 / 12.94 us/op versus close reverted controls at 5.77 / 5.77 / 12.18.
