# Prisma Client performance intermediate report

Date: 2026-06-12.

This report summarizes the current magnitude of the performance and memory gains from the ongoing Prisma Client performance branch. The numbers are from the persistent journal and focused probes in this branch; many rows are microbenchmarks over fake adapters or Miniflare/workerd harnesses, so they should be read as directional product-path evidence rather than final customer benchmarks.

Latest refresh: includes the June 12 Workerd `findFirst users` exact-helper coverage, the query-compiler result-reachability cache through engines commit `63ac52cb5bf`, dependency reload candidate reuse through engines commit `202f7b09aaa`, projected-dependency edge iteration through engines commit `179cdcbbbf1`, the prepared request-surface benchmark plus precomputed-miss safety patch through Prisma commit `73e47489c`, and the accompanying journal/field-note updates.

## Headline

The branch has already found and landed enough product-path work to make several hot generated-client cache-hit rows materially faster, with the largest gains on Cloudflare/workerd simple reads and Node generated-client rows. The 3x target is met or exceeded for some simple/cache-hit paths, but not uniformly for nested generated-client rows yet.

The June 12 call-surface pass kept one additional edge-only product slice: extension-free `wasm-compiler-edge` model delegates now use lazy plain-object properties instead of an inner model composite proxy. Two narrower follow-ups, cached `paramOverrides` metadata and a specialized extension-free edge applied-client proxy, were rejected by close Workerd controls.

The latest accepted exact-helper extension is narrower: flat `findFirst` / `findFirstOrThrow` internal specs now reuse the strict `where + select` descriptor-bound matcher shape. This is productization groundwork, not a broad default feature, but the matching Workerd `findFirst users` row repeated positive for runtime exact helpers. The prepared-operation lead also now has a product-shaped request-surface row, and its cache-miss path compiles from the stored parameterized query instead of reusing stale warmup values.

## Biggest Contributors

1. Generated-client cache-hit/precomputed request path.

This is the largest CPU contributor. The branch proved that generated requests can carry descriptor-derived cache-hit data through the real request path and skip engine-side parameterization and cache-key construction. The product-safe default path now keeps RequestHandler/DataLoader semantics, with guards for transactions, extensions, tracing, debug, SQL commenters, global omit, non-root data paths, and unsupported call shapes.

Notable effects:

- Node default-on request-precomputed A/B moved generated `findUnique` 7.02 -> 4.25 us/op, batched `findUnique` 13.41 -> 8.55, and blog-page 19.86 -> 14.33.
- Workerd request-precomputed rows moved generated `findUnique` 7.50 -> 4.55 us/op worker loop and blog-page 19.20 -> 12.20 in the explicit request-layer mode.
- Direct engine-precomputed rows show a lower bound around 1.5-2.4 us/op for simple Worker hits and 8-10 us/op for nested Worker hits, but those bypass semantics that still need product-safe support.
- A fresh constant-`take` nested `Post.findMany` blog-feed row shows descriptor-bound exact helpers still have product-shaped room on non-batchable nested reads: a corrected 300k Node run measured default / request-precomputed / descriptor-bound static / exact-helper at 22.24 / 24.02 / 19.09 / 19.26 us/op. Workerd coverage for the same shape measured default / engine-precomputed / request-precomputed / descriptor-bound static / exact-helper at 22.18 / 17.79 / 17.76 / 17.77 / 17.95 us/op host dispatch over 10k iterations, with all precomputed rows hitting 10000/10000. This is an about 13% Node exact-helper ceiling and a 19-20% Workerd precomputed-path signal for that feed shape, not a default product win yet. `take` is a literal cache-key component today, so this lead is `take: 10` only unless parameterization changes.
- The stricter `Post.findMany` feed-by-author template adds a real parameterized filter while keeping pagination structural: Node 300k generated default / request-precomputed / descriptor-bound static / exact-helper measured 21.09 / 20.78 / 19.48 / 19.28 us/op. A focused 20k Workerd repeat measured worker-loop default / request-precomputed / descriptor-bound static / exact-helper at 18.75 / 15.85 / 16.55 / 14.85 us/op, with all precomputed rows hitting 20000/20000. This is still an internal explicit template, but it proves the generated-template path can extract a non-pagination placeholder for a nested `findMany` shape without a generic nested matcher.
- The prepared exact operation benchmark pushes that same by-author feed shape further by removing public nested args object construction and descriptor matching after the first cache learn. The first Node 300k run measured direct prepared / request-surface prepared at 7.53 / 7.87 us/op versus exact-helper 9.22, trusted hoisted helper 8.34, cached-wrapper trusted 7.83, and direct/local lower bounds around 7.19-7.20. After making `PrecomputedQueryPlanCacheHit.parameterizedQuery` mandatory and teaching `ClientEngine.requestPrecomputedCachedResult()` misses to compile from that stored query, Node repeated at direct prepared 7.62-7.68 and request-surface 7.79-7.81 us/op, with a no-cache row proving the miss path over 100 iterations. Workerd 20k originally measured worker-loop direct prepared / request-surface prepared at 5.05 / 5.40 us/op versus exact-helper 6.35 and hoisted exact 5.95; the safety refresh measured request-surface at 5.68 us/op over 50k worker-loop iterations. This is benchmark-only evidence, but it makes the next generated-owned call-surface lead concrete while including PrismaPromise, RequestHandler cached-result plumbing, and safe cache-miss semantics.
- Flat `findFirst` / `findFirstOrThrow` exact-helper specs are now covered on both Node and Workerd for exact one-field scalar/enum equality plus exact select. Node 300k `findFirst users` repeats measured request-precomputed / hand exact / runtime exact at `2.25 / 2.07 / 2.08 us/op`, then `2.25 / 2.09 / 2.14`. Workerd 20k repeats measured host dispatch request-precomputed / hand exact / runtime exact at `3.31 / 3.04 / 2.93 us/op`, then `3.24 / 3.01 / 2.93`, with all rows hitting `20000/0`. This is a narrow internal helper signal; `findFirst` remains too broad for generic filter matching without oracle coverage.
- The latest accepted call-surface slice is smaller but target-runtime-positive: edge plain model delegates moved rebuilt 50k Workerd feed-by-author worker-loop descriptor-bound / exact-helper / hoisted exact rows from 7.02 / 6.40 / 6.22 us/op to 6.80 / 6.10 / 5.98. That is roughly 3-5% on the priority nested edge rows, on top of the larger cache-hit and raw-nested wins above.

2. Public API and serializer allocation cleanup.

Before the precomputed path became default, a long sequence of generated-client hot-path cleanups reduced public API overhead: cached model action proxy properties, prototype PrismaPromise methods, empty-extension `_request()` fast path, `queueMicrotask()` DataLoader scheduling, deferred batch-key construction, cheaper selection traversal, lazy serializer path materialization, avoiding rest-object allocation for selection-only fields, omitting empty JSON protocol arguments, cached serializer field tables, and cheap debug/timing guards.

This moved the Workerd generated-client band from roughly 22.50 / 44.40 us/op for simple/nested calibrated rows to roughly 6-7 / 18-20 us/op before the later request-precomputed product path tightened it further.

3. Query-plan cache shape and memory work.

The branch converted many query-plan protocol structures from object-heavy forms to compact tuples/omissions, added string/result-node sharing, made compact raw-nested plans participate in interning, and then replaced retained per-entry refcount `Map`s with compact arrays. The important result is not just smaller serialized plans; retained V8 heap for many cached nested plans dropped substantially.

Big memory wins:

- Raw-nested child subtree/string interning recovered sharing that compact raw-nested emission had initially bypassed: retained blog-page plans dropped from about 16.8 MiB to about 5.3 MiB in the node warm probe.
- Compact interner-count storage reduced restarted-baseline blog-page node warm 4.88 -> 4.55 MiB and parameterized node warm 5.10 -> 4.75 MiB; the later short `@parent$...` scope-string interner moved a fresh node warm baseline 4.54 -> 4.47 MiB and parameterized node warm 4.76 -> 4.69 MiB, with neutral compile-miss timing.
- Workerd retained blog-page plan cache validation shows 100 entries at 335.4 KiB serialized plan bytes with stable generated-client timing.
- The latest compact-only internal-format cleanups removed legacy object-shaped `templateSql`, tagged fragments, legacy query-plan PrismaValue placeholder/generator readers, legacy validation rule/error readers, legacy binding/join support structures, legacy result-object containers, legacy expression-node objects, stale TS-only raw-nested `m` relation readers, stale object-shaped query/join/dataMap walkers in `QueryPlanCache`, JSON-protocol `Param` compatibility in compacted batch responses, raw-nested string column refs, missing raw-nested field-type fallback, canonical scalar-string support in compact template SQL `argTypes`, and canonical result `FieldType` scalar tags from the runtime/types/tests/probes. This is smaller than the earlier memory wins, but it keeps internal hot paths version-lockstep-only. Close direct-plan timing A/Bs over 300k nested blog-page iterations measured 12.62 us/op control vs 10.93 us/op patched for `templateSql`, then 9.05 us/op control vs 8.52 us/op patched for PrismaValue after preserving the monomorphic local-object guard shape; the validation cleanup is not on this cached read path, and the direct-plan smoke stayed in the recent band at 8.54 us/op. The tuned binding/join cleanup measured 8.34 vs 8.36 us/op in a 1M patched/control pair and 8.31 us/op on a 300k confirmation run after direct-indexing hot interpreter paths. The result-object cleanup was neutral on the same direct-plan row: 300k patched/control was 7.07 vs 7.16 us/op, the 1M patched/control pair was 6.51 vs 6.51 us/op, and the 300k reapply smoke was 6.76 us/op. The expression-node cleanup measured 6.87 vs 6.80 us/op in a noisy 300k patched/control pair, then 6.34 vs 6.54 us/op in the 1M patched/control pair, with a 6.71 us/op 300k reapply smoke. The raw-nested `m` cleanup kept the final-owner fast path in band: direct compiled blog-page 5.54 us/op and full generated blog-page 12.12 us/op. The compacted-batch `Param` cleanup kept generated batched `findUnique` healthy at 7.22 us/op. The raw-nested string-ref cleanup used producer proof that 89/89 full-corpus raw-nested mappings were numeric, then measured direct compiled blog-page 5.68 us/op and generated blog-page 11.16 us/op. Requiring raw-nested field types kept direct compiled blog-page at 5.67 us/op and moved generated blog-page to 10.91 us/op. Removing canonical compact SQL arg scalar names stayed in band at 5.56 us/op direct and 10.66 us/op generated on repeats. Removing canonical result `FieldType` scalar tags was kept as lockstep format hygiene, not as a speed win: a close same-session control measured 5.62 us/op direct and 11.40 us/op generated, while the final reapplied rows measured 6.02 and 12.05 us/op. A rebuilt producer/runtime shape scan over sqlite benchmark plans plus PostgreSQL `DataTypes` reported 0 canonical result tags, 1 compact structured field type, and 38 compact raw-nested field types. The cache-memory probe after this cleanup stayed in the recent band: blog-page edge warm retained 100 plans at 693.9 KiB heap delta, 335.4 KiB serialized plan JSON, and 48.3 KiB keys.

4. Raw nested read protocol and runtime.

The branch added compiler-emitted compact raw-nested read (`n`) plans and runtime support that avoids generic `serializeSql()` / `Join` / outer `dataMap` assembly for supported query-mode nested reads. Runtime numeric mapping, local relation scopes, wrapper assembly, relation fanout, and indexed relation attachment each moved parts of the nested path.

Current state:

- Initial local-Wasm blog-page comparisons had direct/local nested execution around 10-11 us/op and compact raw-nested node around 8.5 us/op.
- Current focused compact raw-nested rows are commonly around 4.3-4.5 us/op for the final-owner blog-page shape after compiled query-leaf rendering, with older generic compact shapes still depending on the relation topology.
- A synthetic 100-parent attachment-heavy raw-nested shape improved 208.0 -> 79.7 us/op after thresholded relation indexes.
- Benchmark-only static-wave writer rows previously reached the direct assembler lower bound around Node 3.90-4.06 us/op and Workerd 2.15 us/op worker loop. The accepted compiled query-leaf renderer now brings the product final-owner path much closer to that Node lower bound without changing the serialized raw-nested format.
- A later direct leaf relation shortcut moved compact node 6.57 -> 6.11 us/op and exact compact 6.68 -> 6.30 in a close Node A/B, but was rejected because the 6-7% local win was below the 15% gate and the generated blog-page smoke did not strengthen the product-path signal.
- A serializable compact `N` writer-program node with strict unique-root numeric payloads also stayed too generic: it measured 5.92-6.03 us/op versus compact `n` at 6.49-6.97 and the static-wave lower bound at 3.93-4.06, with higher heap churn.
- The first product-shaped final-owner writer path is now kept. It targets the current direct-only raw-nested relation shape for unique root reads with direct unique children, a wrapper-list child, and a list child with a unique nested child. A serial side-worktree A/B moved `direct plan after phase warmup blog page / nested rows` 7.37 -> 5.73 us/op, `generated client exact descriptor helper blog page` 11.21 -> 10.89, and full `generated client blog page` 12.96 -> 11.87. Main-checkout confirmation after porting measured 5.65, 10.52, and 11.87 us/op respectively. This still leaves headroom to the 3.8-4.0 us/op static-wave writer, but it is the first accepted runtime step that changes final-owner write ownership rather than only flattening generic execution.
- The Workerd refresh after the final-owner writer and stale raw-nested `m` deletion measured default generated blog-page at 11.50 us/op host-dispatch upper-bound over 20k iterations, request-precomputed at 11.04, descriptor-bound static at 10.81, engine-precomputed at 8.12, and static-wave writer lower bound at 1.82. The worker-internal request loop for the default generated blog-page row was 10.30 us/op. This confirms the nested path stays healthy on the Cloudflare-like harness, while the mixed descriptor rows also show that productization still needs stable-shape and alternating-shape gates rather than trusting one helper mode.
- The later raw-nested string-ref, missing-field-type, compact SQL arg-type, and result `FieldType` canonical-tag compatibility deletions did not require a Workerd rebuild; Node focused rows stayed in the recent band at 6.02 us/op for direct compiled blog-page and 12.05 us/op for the full generated blog-page row after the result `FieldType` cleanup. A following small raw-nested scope-dedupe cleanup avoided `Set` allocation for small fanouts and measured 5.78 us/op for direct compiled blog-page, 5.35 us/op for the exact compact-node row, and 11.77 us/op for the full generated blog-page row.
- The current compiled final-owner query-leaf renderer is the next accepted product-shaped step. It compiles strict local-scope compact template SQL leaves once per final-owner relation query, supporting direct placeholders/constants and `T` tuple fragments while falling back to generic rendering for unsupported shapes or max-bind overflow. This removed both relation scope-object construction and generic render/evaluate machinery for the blog-page final-owner relation queries. Close controls in the same turn measured direct / exact compact / generated nested rows at 5.77 / 5.77 / 12.18 us/op; the kept rows measured 4.45 / 4.39 / 9.96, with higher-iteration repeats at 4.34 direct and 9.53 generated. A properly warmed Workerd 20k nested smoke measured default generated blog-page at 11.50 us/op host-dispatch and 10.40 us/op worker loop, request-precomputed at 10.15 / 9.10, and engine-precomputed at 7.63 / 6.45. This keeps the current direction aligned with the internal lockstep rule and the final-owner ownership direction: replace formats outright when producer evidence proves the old reader is unreachable, and remove concrete per-query overhead without reintroducing broad generic execution scaffolding.
- A fresh benchmark-only non-unique blog-feed direct/raw-nested row gives useful lower-bound coverage, but not the immediate product runtime target. The 100k Node run measured generated default / request-precomputed / descriptor-bound static / exact-helper at 23.19 / 21.69 / 19.18 / 18.21 us/op, direct/local execution at about 15.7-15.8 us/op, and the benchmark-only compact raw-nested node at 9.34-9.36 us/op. A follow-up non-unique `n` final-owner spike improved that synthetic compact-node row to about 6.8-7.0 us/op, but close product rows did not improve because the real `Post.findMany` feed plan currently compiles to compact joins (`d/l/j`), not raw-nested `n`. The next feed proof should target the current compact join product plan or change compiler emission first, then rerun generated/direct rows.

A 300k adjacent Node A/B moved `direct plan blog feed / nested rows` 15.55 -> 13.67 us/op and `generated client blog feed / nested rows warmed cache` 20.84 -> 18.81 us/op. The runtime intentionally has no seven-slot relation compatibility reader; producer and consumer are version-lockstep.

It builds root owner records directly, runs relation queries with page-level scopes, filters supported relation-op child rows before loading nested unique children, and returns root arrays in query order. Close 100k A/B moved generated default / request-precomputed / exact-helper from 12.90 / 12.64 / 11.30 us/op to 10.31 / 10.62 / 9.31, and direct plan from 9.28 to 7.40. This replaces the earlier "non-unique final-owner is benchmark-only" conclusion; it became product-relevant only after compiler relation-ops emission made the feed compile to raw-nested.
- The new feed-by-author exact cached-wrapper lower-bound row measured generated exact-helper at 9.74 us/op, cached-wrapper exact descriptor at 8.80, and cached-wrapper lazy descriptor at 10.90 in the same 100k Node run. That puts about 0.9 us/op between the current product exact-helper path and the cached-wrapper exact lower bound; it also confirms the lazy descriptor shape is not the right lower-bound comparator for generated exact-helper work.
- A follow-up trusted author-id-only descriptor-helper row measured the ceiling for skipping full nested shape validation after a hypothetical generated-shape proof. The 300k generated full exact-helper / trusted-helper rows were 9.10 / 8.36 us/op, and cached-wrapper exact / trusted rows were 8.20 / 7.65. This is benchmark coverage only; the current exact matcher must keep full user-arg shape validation.
- The latest accepted call-surface product slice is edge-only. `wasm-compiler-edge` extension-free model delegates now use lazy plain-object properties instead of an inner model composite proxy, while Node `client` and extension-bearing delegates keep the old proxy path. 5. Rust query-compiler allocation and compile-time work.

The largest accepted Rust-side win is changing M:N nested reads from `dataMap` / join plans to compact raw-nested reads where guardrails allow it:

- `query-m2m` full compile allocations 941 -> 797 and Criterion 40.625 -> 36.055 us.
- `query-many-m2m` full compile allocations 912 -> 795 and Criterion 40.831 -> 36.203 us.
- M:N-containing connectOrCreate rows saved roughly 300 full-compile allocations and improved about 9-10%.

Smaller Rust allocation wins were kept only when Criterion stayed neutral-to-positive: selection-order consuming passes, parser path mutable stack, `SmallVec` storage for that stack, raw nested relation scalar helpers, borrowed related-read compilation, placeholder scalar serialization, direct `ModelProjection::as_columns()` collection, and the no-search grouped-filter merge guard. The latest filter guard saved four graph-build/full-compile allocations on `filter-not-contains-param` while scalar controls stayed allocation-neutral; close Criterion plus a reversed-control repeat was timing-neutral. The allocation profiles still point at `graph_build` and `translate_ir` mid-sized owned structures, not a broad `Arc` clone problem.

The latest kept Rust-side cleanup is another small but clean graph-build win: engines commit `8778c84c831` streams nested relation linking-field selections through `FieldSelection::union_iter()` instead of collecting a temporary `Vec`. It saves one `graph_build` and `full_compile` allocation/op on the sampled nested read/write fixtures. The same-session Criterion control was positive across the focused filter; representative patched vs reverted medians were `query-m2o` 28.13 vs 30.84 us, `query-many-m2m` 33.68 vs 36.60 us, `nested-pagination-query` 30.10 vs 33.55 us, and `update-set-nested` 101.78 vs 107.07 us.

The newest Rust-side allocation win replaces `ResultNode::Object`'s ordered `IndexMap` with a capacity-sized `Vec` in the query compiler result mapper. This is different from the earlier rejected "pre-size the IndexMap" spike: the kept patch removes hash-table storage entirely while preserving the same serialized result-object map. Close allocation profiles saved `translate_ir` bytes across representative rows, including `query-m2o` 27.4 -> 26.0 KiB, `query-many-m2m` 34.0 -> 32.6 KiB, `aggregate-custom` 33.0 -> 31.2 KiB, and `create-nested-connectOrCreate-mixed` 149.3 -> 147.8 KiB. Criterion improved sampled compile rows by about 2-6% (`query-many-one2m` -6.0%, `aggregate` -5.4%, `aggregate-custom` -4.9%, `query-m2o` -3.2%, `update-set-nested` -2.3%), while `nested-pagination-query` was neutral.

## What Did Not Pay Off

Several plausible ideas were tested and rejected:

- `serde_wasm_bindgen` / one-pass JSON deserialization as a standalone path is unlikely to be the main win. The native allocation profile shows JSON parse/adaptation is smaller than `graph_build` and `translate_ir`, and cache hits still need JS-side cache-key/shape work anyway. A 2026-06-11 code scout confirmed the current request path still has no active `compileFromValue` entrypoint and that a `serde_wasm_bindgen::from_value()` replacement would still build owned `JsonBody` / `IndexMap<String, serde_json::Value>` / `Operation` / `ArgumentValue` trees before graph build.
- A direct `compileFromValue(JsValue)` Wasm entrypoint is also too shallow if it still builds today's owned Rust protocol maps. The 2026-06-10 release-Wasm spike measured current string compile misses versus JS-value compile misses at 404.42 vs 422.08 us/op for `findUnique`, 299.61 vs 300.55 for `findMany`, and 2306.57 vs 2348.81 for blog-page, so it was reverted.
- Cache-key-only work is low ceiling on Workerd: pure cache-key/lookup is about 0.4 us/op for `findUnique` and 1.5 us/op for blog-page, far below the full generated-client cost.
- Generic descriptor matchers, generic raw-nested flat schedules, and instruction-array executors that still allocate generic result/record/scope/slot arrays do not beat the current paths enough to productize.
- Cache-handle-only nested work is too small by itself. A fresh 2026-06-10 300k nested blog-page run measured `cached request wrapper static shape` at 6.92 us/op versus direct plan at 6.55 us/op, while `cached request wrapper static descriptor` was 8.10 and lazy descriptor was 9.42. Simple Worker cache-hit paths are already past 3x, and the latest 20k Workerd host-dispatch refresh puts nested generated blog-page around 3.9x faster than the early calibrated baseline. Internal engine-precomputed rows and static-wave raw-writer lower bounds still show enough headroom to plausibly reach the remaining target if the next product shapes preserve semantics.
