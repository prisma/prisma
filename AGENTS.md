# Prisma Repository – Agent Field Notes

> **Meta note**: This is the primary agent knowledge base file. `CLAUDE.md` and `GEMINI.md` are symlinks to this file—always edit `AGENTS.md` directly. When learning something new about the codebase that would help with future tasks, update this file immediately.

- **Repository scope**: Monorepo for Prisma ORM, CLI, client, tests, etc. Many packages use TypeScript, Rust and WebAssembly (via engines), TSX, and Jest; automation relies on `pnpm`. Expect large fixture directories and generated files.
- **Workspace layout**: Managed via pnpm workspaces and Turborepo (`turbo.json`). Node ^20.19 || ^22.12 || >=24.0 and pnpm >=10.15 <11 are required (see root `package.json`). Top-level scripts (`pnpm build`, `pnpm dev`, `pnpm test`) delegate into `scripts/ci/publish.ts`; package-specific commands run with `pnpm --filter @prisma/<pkg> <script>`. Turborepo caches builds; run `pnpm build` from root to build all packages in dependency order.
- **Key packages**: `packages/cli` (Prisma CLI entry point), `packages/migrate` (migrate/db namespace + fixtures), `packages/client` (client runtime), `packages/client-generator-ts` (new `prisma-client` generator), `packages/client-generator-js` (traditional `prisma-client-js` generator), `packages/client-generator-registry` (generator registry for managing generators), `packages/client-engine-runtime` (the core part of the new Rust binary free client based on Wasm query compiler, used by `ClientEngine` class in `packages/client`), `packages/client-common` (shared client utilities), `packages/client-runtime-utils` (utility types and singletons for Prisma Client), `packages/config` (`PrismaConfigInternal` + loader), `packages/internals` (shared CLI + engine glue), `packages/engines` (Rust binaries download wrapper), `packages/integration-tests` (matrix suites), `packages/query-plan-executor` (standalone query plan executor service for Prisma Accelerate), `packages/credentials-store` (credential storage utilities), sqlcommenter packages under `packages/sqlcommenter*`, and numerous driver adapters under `packages/adapter-*`.
- **Driver adapters & runtimes**: `packages/bundled-js-drivers` plus the `adapter-*` packages ship JS driver adapters: `adapter-pg`, `adapter-neon`, `adapter-libsql`, `adapter-planetscale`, `adapter-d1`, `adapter-better-sqlite3`, `adapter-mssql`, `adapter-mariadb`, `adapter-ppg` (Prisma Postgres Serverless). These are built on helpers in `driver-adapter-utils`; migrate/client fixtures exercise them, so adapter changes typically require fixture/test updates. CI tests driver adapters with flavors: `js_pg`, `js_neon`, `js_libsql`, `js_planetscale`, `js_d1`, `js_better_sqlite3`, `js_mssql`, `js_mariadb`, `js_pg_cockroachdb`.
- **Build & tooling**: Typescript-first repo with WASM/Rust assets (downloaded by `@prisma/engines`). Multiple `tsconfig.*` drive bundle vs runtime builds. Lint via `pnpm lint`, format via `pnpm format`. Maintenance scripts live in `scripts/` (e.g. `bump-engines.ts`, `bench.ts`, `ci/publish.ts` orchestrates build/test/publish flows). Build configuration uses esbuild via `helpers/compile/build.ts` with configs in `helpers/compile/configs.ts`. Most packages use `bundledConfig` which bundles to both CJS and ESM with type declarations.
- **Benchmarking**: Comprehensive performance benchmarks using Benchmark.js with CodSpeed integration. See `docs/benchmarking.md` for full documentation. Key commands:
  - `pnpm bench` - Run all benchmarks (outputs to `output.txt`)
  - `pnpm bench <pattern>` - Run benchmarks matching pattern
  - Benchmark locations:
    - End-to-end query performance: `packages/client/src/__tests__/benchmarks/query-performance/query-performance.bench.ts`
    - Query compilation/cache behavior: `packages/client/src/__tests__/benchmarks/query-performance/caching.bench.ts`
    - Query interpreter/data mapper: `packages/client-engine-runtime/bench/interpreter.bench.ts`
    - Query plan cache memory probe: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/query-plan-cache-memory.ts`
    - Local `ClientEngine` cache-hit timing probe with empty SQLite adapter: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/client-engine-cache-timing.ts`
    - Miniflare/workerd query compiler memory probe: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/workerd-query-compiler-memory.ts`
    - Client generation: `packages/client/src/__tests__/benchmarks/huge-schema/`, `packages/client/src/__tests__/benchmarks/lots-of-relations/`
  - Benchmarks run automatically on CI via `.github/workflows/benchmark.yml`; CodSpeed tracks performance over time and alerts on >100% regression.
  - The ongoing Prisma Client performance investigation journal is `wip/client-performance-journal.md`; update it after each accepted change, rejected experiment, measurement, and new lead so work survives context compactions and harness restarts.
  - `client-engine-cache-timing.ts` supports `CLIENT_ENGINE_CACHE_TIMING_FILTER=<substring>` to run only matching rows and `CLIENT_ENGINE_CACHE_TIMING_ITERATIONS=<n>` to override row iterations. This is useful for CPU profiling one nested cached-plan row, e.g. `CLIENT_ENGINE_CACHE_TIMING_FILTER='direct plan after phase warmup' CLIENT_ENGINE_CACHE_TIMING_ITERATIONS=50000 node --cpu-prof --import tsx packages/client/src/__tests__/benchmarks/query-performance/client-engine-cache-timing.ts`.
  - For Cloudflare/workerd probes, instantiate query compiler Wasm through a static module import / Miniflare `CompiledWasm` module. Dynamic `new WebAssembly.Module(bytes)` inside the worker fails with `Wasm code generation disallowed by embedder`. When importing `packages/client/runtime/wasm-compiler-edge.mjs` in Miniflare, explicitly register the built `@prisma/client-runtime-utils` ESM under that bare module specifier. Keep manual compiler and generated-client rows in separate Miniflare instances because both instantiate the wasm-bindgen query compiler runtime. Workerd's worker-internal timer can be too coarse for warmed pure-JS request loops; use host `dispatchFetch` timing as an upper-bound signal.
  - For query-compiler graph traversal micro-optimizations, require Criterion evidence in addition to allocation counts. Direct `petgraph` iteration and helper-level `SmallVec` experiments around `subgraph_contains_result()`, `is_direct_child()`, and `direct_child_pairs()` either had no allocation effect or regressed/failed to improve nested-create benches despite saving one or two allocations.
  - For query-compiler translation allocation work, removing expression scaffolding that `Expression::simplify()` would discard can be worthwhile only with timing proof: the accepted single-result-subgraph fast path in `fold_result_scopes()` saved four nested-create allocations and improved the focused rerun. In contrast, rewriting `process_children()` result-position splitting saved one allocation and returning binding-only child `Let` nodes directly from `process_child_with_dependencies()` saved two allocations, but both regressed nested-create Criterion.
  - For nested cached-plan runtime work, skipping `klona()` for static `InMemoryOps` in compact `p` / legacy `process` nodes was rejected: scan-only and WeakMap-cached static/dynamic guards did not clearly improve cached wrapper, inner-plan, or phase-warmed direct-plan nested rows.
  - For `QueryInterpreter` query leaves, guarded one-rendered-query fast paths are viable: legacy `query` / `execute` and compact `q` / `x` can bypass the generic rendered-query loop when SQL comments and query instrumentation are inactive. This improved/kept healthy the nested-row direct/local/inner timing rows and the standalone interpreter bench.
  - For compact `QueryInterpreter` nodes, the per-interpreter compiled-node WeakMap is an accepted cached-plan win. It compiles hot compact tags (`v`, `s`, `g`, `l`, `e`, `q`, `x`, `u`, `m`, `j`, `d`, `p`) into closures, preserves the direct `d(q)` / `d(u(q))` data-map fast paths, and falls back to the old compact interpreter for unsupported tags.
  - `QueryPlanCache` keeps a last-hit entry for single and batch lookups to avoid repeated `Map.get()` calls on warmed same-shape cache hits. Preserve the eviction invalidation tests if touching this path, because stale last-hit entries can otherwise bypass the LRU maps after eviction.

The legacy `pnpm --filter @prisma/client test` command primarily runs the older Jest unit tests plus `tsd` type checks. Migrate CLI suites live in `packages/migrate/src/__tests__`, the CLI runs both Jest (legacy suites) and Vitest (new subcommand coverage) via `pnpm --filter prisma test`, and end-to-end coverage lives in `packages/integration-tests`.

- **Client architecture (Prisma 7)**:
  - `ClientEngine` in `packages/client/src/runtime/core/engines/client/` orchestrates query execution using Wasm query compiler.
  - `ClientEngine` query plan cache defaults are build-target specific: Node.js `client` builds keep up to 1000 entries, while `wasm-compiler-edge` builds keep up to 100 entries by default to reduce retained memory. `queryPlanCacheMaxSize: 0` disables the cache.
  - `ClientEngine.request()` and `requestBatch()` use a synchronous connected-engine fast path before falling back to async `#ensureStarted()`; keep that path when editing hot request flow because it avoids an avoidable async hop on warmed clients.
  - Keep disabled debug logging out of `ClientEngine` warmed request hot paths. `@prisma/debug` records call arguments in a global history even when the namespace is not enabled, so unguarded per-request `debug(...)` calls can retain query plans and add microseconds of overhead.
  - `QueryPlanCache` interns repeated string values only for join-shaped plans and only for strings with length >= 8. The interner is refcounted per cache entry so updates, evictions, and `clear()` release strings from old plans; avoid broad all-plan string interning because scalar-selection plans mostly have unique SQL strings and can regress from bookkeeping overhead.
  - Do not split cached `templateSql.fragments` strings inside `QueryPlanCache` just to expose common SQL suffixes or selected-column chunks for interning. A local spike splitting long strings at `FROM` and then at comma boundaries passed cache tests but did not reduce retained V8 heap for blog-page plans and worsened larger 1,000-shape rows.
  - `client-engine-cache-timing.ts` has cached request wrapper rows using real `QueryPlanCache` + `LocalExecutor`, plus a scripted nested blog-page adapter that returns one post, author/category, two tags, and two comments/comment authors (seven SQL calls per request). After the disabled-debug guard, generic `ClientEngine` wrapper glue is a small part of warmed cache-hit cost; bigger remaining targets are cache-key/parameterization and cached-plan execution/rendering.
  - `client-engine-cache-timing.ts` repeats the full `ClientEngine` blog-page nested warmed-cache row after the phase-split rows. Prefer this late row when judging full-engine overhead: local runs showed the early warmed row was order/JIT sensitive (about 68-76 us/op early vs about 52-57 us/op late), while the late row stayed about 8 us/op above cached request wrapper rows.
  - Do not add a narrow connected/non-debug/no-sql-commenter single-query cache-hit fast path to `ClientEngine.request()` without new evidence. A local spike bypassed the generic request path on cache hits, but late full-engine nested rows stayed around 52 us/op and did not shrink the gap to cached request wrapper rows.
  - Do not cache `ClientEngine`'s `hasSqlCommenters` check as a private constructor field without stronger evidence. A local spike built cleanly, but same-command late warmed nested rows were slightly faster with the legacy per-request config check (17.46 us/op legacy vs 17.61 us/op cached field over 100k iterations).
  - `client-engine-cache-timing.ts` also has render-only, data-map-only, and nested root-data-map phase rows for cached plans. Current rows show `renderQuery()` for the first DB query and scalar data mapping are small relative to nested cached-plan execution. The nested root `dataMap` split is sensitive to JIT/order effects: repeated "direct plan after phase warmup" rows dropped from roughly 57-58 us/op to roughly 45 us/op, close to manual inner+outer, so do not treat root `dataMap` pushdown as proven without warmup-controlled measurements.
  - Exact-shape cache-key phase rows in `client-engine-cache-timing.ts` show `parameterizeQuery()` dominates cache-key cost; `JSON.stringify()` plus key concatenation is comparatively small on current Node/V8 measurements.
  - Do not switch single-query cache keys to the full compile request string without new evidence. A local `request as cache key` benchmark row was slower on simple shapes, retained larger keys, and was only noise-level on the nested blog-page shape.
  - Non-compacted `multi` batch query plans can also populate the single-query plan cache when the individual plans fit within the configured cache size. This lets later matching single queries reuse plans compiled as part of a mixed batch. Compacted batches are intentionally not decomposed.
  - `QueryPlanCache.getSingle()` / `getBatch()` have a last-hit fast path for repeated same-shape cache hits; they still call the LRU touch helper, and evictions clear matching last-hit pointers. `QueryPlanCache` also performs cache-local, refcounted sharing for retained compact join plans: repeated strings are interned, repeated child `q`/`x` database-query templates under joins are canonicalized across cached entries, and repeated nested result structures under `dataMap` nodes are shared while root result structures remain distinct. Root query templates are intentionally not canonicalized so distinct root SQL strings do not bloat the interner with one-off large keys.
  - `QueryPlanCache` only interns strings with length >= 32. This keeps repeated SQL fragments and larger field metadata shared while avoiding bookkeeping for many short strings. A local cache-memory A/B showed lowering the threshold to 1 worsened retained heap, and skipping root `q`/`x` query-template string interning also worsened retained blog-page plan caches.
  - `packages/client/src/__tests__/benchmarks/query-performance/workerd-query-compiler-memory.ts` has raw compile-retention scenarios, `mode=client-cache` scenarios that manually parameterize benchmark value churn, and `mode=client-execute` scenarios that instantiate a generated-like edge Prisma Client with a fake SQLite adapter to execute warmed cache-hit requests in Miniflare/workerd.
  - In `client-engine-cache-timing.ts`, the fake blog-page adapter's `getBlogPageResultSet()` caches resolved `SqlResultSet`s by rendered SQL string and falls back to parsing the table name from the SQLite `FROM main.<table>` fragment; keep that shape when editing the benchmark. The older repeated `sql.includes(...)` classifier added noticeable benchmark-only overhead (about 1.86 us/op adapter-only vs about 0.79 us/op with the SQL-result cache), so treat adapter classifier samples as harness overhead rather than product runtime.
  - Two executor implementations: `LocalExecutor` (driver adapters, direct DB) and `RemoteExecutor` (Accelerate/Data Proxy).
  - `LocalExecutor.execute()` keeps the warmed non-transaction path free of its own `async` frame and reuses stable interpreter transaction-manager option objects. If editing this path, preserve the split transaction helper unless new benchmarks show it is no longer useful.
  - `QueryInterpreter` class in `packages/client-engine-runtime/src/interpreter/query-interpreter.ts` executes query plans against `SqlQueryable` (driver adapter interface).
  - `QueryInterpreter` compiles hot compact tuple nodes once per plan object into closures cached in a per-instance `WeakMap`. This avoids redispatching through `#interpretCompactNode` on cached-plan hot paths while keeping fallback behavior for less common tags (`c`, `+`, `R`, `r`, `t`, `V`, `?`, `0`, `-`, `i`, `M`).
  - `QueryInterpreter.run()` precomputes per-run SQL-commenter and query-instrumentation booleans in the runtime context; avoid rechecking those flags inside each query node on cached-plan hot paths.
  - `QueryInterpreter.run()` uses outer non-raw driver-error mapping, while raw SQL query/execute paths keep local `rethrowAsUserFacingRawError` handling so `$queryRaw`/`$executeRaw` still surface P2010. The common non-raw single-query `dataMap(query)` path skips the result-set merge loop when SQL commenters and query instrumentation are disabled.
  - `QueryInterpreter` join nodes avoid `Promise.all(...map(async ...))` when there is exactly one child join expression. The blog-page cached plan has join child counts `[4, 1, 1]`, so this mainly helps nested joins with actual child rows; empty-row ClientEngine timing rows can stay noisy.
  - Do not collapse `QueryInterpreter` `let`/`l` scope handling to a single reused nested context just to avoid repeated object spreads; that benchmarked neutral-to-soft on the current cached-plan probes.
  - Query flow: `PrismaClient` → `ClientEngine.request()` → query compiler → `executor.execute()` → `QueryInterpreter.run()` → driver adapter.
  - Local `ClientEngine` execution opts `QueryInterpreter` into native JS result values and marks non-raw responses so `RequestHandler` skips JSON-protocol deserialization; `RemoteExecutor` and `packages/query-plan-executor` keep JSON-protocol-shaped values and must not skip normalization/deserialization.
  - `QueryInterpreter` precomputes its max SQL bind/chunk size once from `ConnectionInfo.maxBindValues` or provider defaults and passes the field to `renderQuery()` from query leaves. Keep the adapter override precedence if editing this path.
  - Query-plan result field nodes inside object `fields` may omit `type: 'field'` and may omit `dbName` when it is identical to the result field key; `packages/client-engine-runtime/src/interpreter/data-mapper.ts` recognizes field nodes by `fieldType` and defaults missing `dbName` to that key. Explicit `dbName` is still required when SQL column names differ from the result field key.
  - `data-mapper.ts` caches result-set field mappings by result fields and column shape, including scalar type name and list arity, so row mapping does not rediscover that metadata per cell. Keep the cached mapping shape guarded by interpreter and product-path timing benchmarks.
  - Do not add a separate field-only result-set mapper path or return existing `Date` objects directly from JS-mode datetime mapping without new benchmark evidence; both were mixed/noisy on nested blog-page rows and did not clearly improve local executor timings.
  - Do not cache `attachChildrenToParents()` join key metadata in a WeakMap without new evidence; it regressed the dedicated join/deep-nested interpreter microbench and did not clearly improve nested blog-page product rows.
  - Do not replace multi-child join `Promise.all(joinExpressions.map(async ...))` with a helper that first builds child promises and then result objects via explicit loops. A local spike passed tests/build and kept interpreter benches in range, but product-shaped nested rows were not positive.
  - Do not split compact `j` execution into direct `attachChildToParents()` calls just to avoid child wrapper objects without new evidence. A local spike slightly improved some inner/local rows but stayed neutral-to-negative on warmed `ClientEngine` and direct nested rows after rebuild, so it was reverted.
  - Do not cache compact mapped-join / nested-single-child-join matcher results in WeakMaps without new evidence. A local spike improved some product-shaped cached-plan rows, but regressed interpreter `join (1:N)`, `sequence`, and `deep nested join` throughput badly enough to revert.
  - Do not rewrite compact mapped-join / nested-single-child-join matchers into direct private execution helpers just to avoid matcher result allocations. A local spike passed focused tests/build, but regressed cached request wrapper nested blog rows to roughly 49-54 us/op versus the current roughly 42 us/op band.
  - Do not skip `cloneObject()` / `evaluateProcessingParameters()` for static-looking `InMemoryOps` with a WeakMap "needs evaluation" guard without new evidence. A local spike improved the isolated tags branch but regressed full nested blog rows and the deep nested interpreter bench, so it was reverted.
  - Do not add a generic no-op pagination guard to `processManyRecords()` just to skip linked pagination when `take >= records.length` without new evidence. A local spike was mixed and regressed deep nested interpreter throughput plus one local executor nested product run, so it was reverted.
  - Current `client-engine-cache-timing.ts` nested blog-page inner plans have no compact `p` process nodes. A benchmark-only attempt to strip process nodes failed with zero nodes found, so do not chase `processRecords()` for that benchmark shape without first revalidating the compiled plan.
  - Do not route generic `query` / compact `q` interpreter nodes through `#executeQuery()` just to reuse the single-query fast path. A local source run was neutral, but after rebuilding `@prisma/client-engine-runtime` it regressed product-shaped nested blog rows to roughly 65 us/op cached wrapper and 61 us/op local executor.
  - Do not inline a no-comment/no-instrumentation single-query fast path inside compact `q` nodes without new evidence. A local spike avoided the generic result-set merge loop for nested child queries, but warmed `ClientEngine` nested rows did not improve and cached request wrapper nested rows regressed to roughly 61-62 us/op.
  - Do not add a compact `u(q)` fast path that calls `#executeQuery()` directly without new evidence. A local spike improved direct nested-plan rows to roughly 37 us/op, but regressed cached request wrapper nested rows to roughly 46.5-46.7 us/op versus the current roughly 42-43 us/op band.
  - Do not specialize `serializeSql()` with computed-key object literals for 0-5 column result rows without new evidence. A local spike worsened direct warmup nested rows to roughly 50 us/op and did not clearly improve cached wrapper/local executor product rows.
  - `serializeSql()` does have accepted unrolled assignment paths for 0-5 columns and exactly 11 columns. This is intentionally not the rejected computed-key object-literal shape; it hoists column names once per result set and assigns into plain row objects. The current `client-engine-cache-timing.ts` / `workerd-query-compiler-memory.ts` fixtures use result-set widths 0, 2, 3, 5, and 11, so verify a new hot width exists before adding more exact serializer cases. Keep `serialize-sql.test.ts`, `data-mapper.test.ts`, `interpreter.bench.ts`, and `client-engine-cache-timing.ts` coverage if editing it.
  - Do not add a separate `renderSingleQuery()` helper just to avoid `renderQuery()`'s one-element `SqlQuery[]` allocation on the no-comments/no-instrumentation fast path. A local A/B over cached request wrapper rows was neutral on cheap rows and slower on nested blog rows; the existing flat template rendering cache plus single-query interpreter fast path is better on current V8.
  - The benchmark blog-page query currently compiles to one outer `dataMap` over a `let`/`join` tree. Nested `q` nodes serialize SQL result sets to row objects first, and the outer data map then remaps the joined object graph into the final response shape. Future nested-row wins likely need data-map pushdown / plan-shape changes rather than tiny `attachChildrenToParents()` key tweaks.
  - `client-engine-cache-timing.ts` includes nested blog-page phase rows for adapter-only `queryRaw` over the seven benchmark result sets, `serializeSql()` over those same result sets, `renderQuery()` over all seven query leaves with representative scopes, an inner-plan variant where compact `q` leaves are replaced with precomputed fresh row objects, per-root-child-branch rows, a variant where the root compact `j` join's immediate children are precomputed, and a variant where the root compact `j` join is replaced with a precomputed fresh joined object. After the compact nested-branch/root mapped-join fast paths and exact-width `serializeSql()` paths, current local rows put all-leaf `renderQuery()` around 1.1 us/op, adapter-only seven result sets around 2.1 us/op, `serializeSql()` over those result sets around 1.75 us/op, precomputed query leaves around 7.6 us/op, precomputed root join children around 4.7 us/op, and precomputed join leaves around 4.6 us/op. Per-child branch rows put simple author/category branches around 4.5 us/op and deeper tags/comments branches around 5.9-6.0 us/op.
  - `data-mapper.ts::mapObjectWithMappings()` avoids `Object.hasOwn()` on the common present, non-`undefined` field/object mapping path and only falls back to ownership checks for `undefined`. This benchmarked as a meaningful nested blog-page outer-data-map win; preserve missing-field behavior for explicit `undefined` properties if touching this path.
  - `QueryInterpreter` has a compact mapped-join fast path for the root compiler shape `l(@parent = q) -> l(@parent$field... = m(field, g @parent)) -> j(g @parent, children...)`. This targets root relation joins such as blog-page author/category/tags/comments and benchmarked as a major nested-row and value-scope win; keep focused coverage in `query-interpreter.test.ts`.
  - `QueryInterpreter` has a compact nested-branch fast path for the compiler shape `l(@parent = q) -> l(@parent$field = m(field, g @parent)) -> j(g @parent, one child)`. This targets nested relation branches such as blog-page `tags` / `comments` and benchmarked as a meaningful nested-row win; keep focused coverage in `query-interpreter.test.ts`.
  - `QueryInterpreter` intentionally reuses one scoped runtime context per legacy/compact `let` scope and compact join fast-path scope instead of cloning `{ ...context, scope }` for every binding or join child. This benchmarked as a small neutral-to-positive cached nested-plan cleanup; preserve the one-context-per-scope shape when touching scope semantics.
  - Do not replace `QueryInterpreter.run()`'s `{ ...options, ...runtimeFields }` context construction with an explicit object copy without stronger evidence. A local spike slightly improved the full cached-wrapper nested blog row but regressed direct/local nested-plan rows materially; the spread shape appears better for the inner interpreter path on current V8.
  - `QueryInterpreter`'s local `mapField()` helper handles array-of-records inline and only recurses for nested arrays. This targets hidden join-key extraction for nested branch arrays and benchmarked as a small neutral-to-positive cleanup; keep the behavior for primitives (`mapField([1], 'id')` returns `[1]`) unless changing the interpreter contract deliberately.
  - Do not add a separate strict single-parent join attachment helper in `attachChildrenToParents()` without new evidence. A local spike that avoided one-element parent array wrapping and repeated parent `asRecord()` calls passed tests but worsened cached wrapper/direct/local nested blog rows, so the generic tiny strict-key helper remains better on current V8.
  - Do not split the common one-child join path into a separate `attachChildToParents()` helper just to avoid the temporary one-element `children` array/object. A same-session A/B after V8 warmup showed patched and unpatched nested blog rows were effectively identical (cached wrapper about 17.7-17.8 us/op, direct/local about 11.9-12.1 us/op), so the extra branch/helper is not worth keeping.
  - Do not change DateTime JS result cloning in `data-mapper.ts` from `new Date(value)` to `new Date(value.getTime())` without new evidence. The clone-preserving patch passed mapper tests and lint, but nested blog rows stayed in the same warm band (cached wrapper roughly 17.6-17.9 us/op, direct/local roughly 11.8-12.1 us/op) and the isolated outer data map did not move.
  - The compiler creates the root data-map wrapper in `/home/aqrln.guest/prisma-engines/query-compiler/query-compiler/src/translate.rs`: `map_result_structure(&graph, ...)` runs before node translation and wraps the translated root sequence in `Expression::DataMap`. Query-mode nested relation plans are built in `translate/query/read.rs` (`add_inmemory_join()` / `build_read_related_records()`), while `core/src/query_graph_builder/read/utils.rs::merge_relation_selections()` injects hidden linking fields needed for `attachChildrenToParents()`. Any child-side data-map/projection pushdown must preserve those join keys until attachment is done.

- **Driver adapter error handling**:
  - Database errors are mapped in each adapter's `errors.ts` (e.g., `packages/adapter-pg/src/errors.ts`).
  - `MappedError` type in `packages/driver-adapter-utils/src/types.ts` defines all known error kinds.
  - Known error kinds include: `GenericJs`, `UnsupportedNativeDataType`, `InvalidIsolationLevel`, `LengthMismatch`, `UniqueConstraintViolation`, `NullConstraintViolation`, `ForeignKeyConstraintViolation`, `DatabaseNotReachable`, `DatabaseDoesNotExist`, `DatabaseAlreadyExists`, `DatabaseAccessDenied`, `ConnectionClosed`, `TlsConnectionError`, `AuthenticationFailed`, `TransactionWriteConflict`, `TableDoesNotExist`, `ColumnNotFound`, `TooManyConnections`, `ValueOutOfRange`, `InvalidInputValue`, `MissingFullTextSearchIndex`, `SocketTimeout`, `InconsistentColumnData`, `TransactionAlreadyClosed`, and database-specific kinds (`postgres`, `mysql`, `sqlite`, `mssql`).
  - `convertDriverError()` in each adapter maps database-specific error codes to `MappedError` kinds.
  - `rethrowAsUserFacing()` in `packages/client-engine-runtime/src/user-facing-error.ts` maps `MappedError` kinds to Prisma error codes (P2xxx).
  - To add a new error mapping: (1) add kind to `MappedError` in driver-adapter-utils, (2) map database error code in relevant adapter(s), (3) add Prisma code mapping in `getErrorCode()` and message in `renderErrorMessage()`.
  - Raw queries (`$executeRaw`, `$queryRaw`) use `rethrowAsUserFacingRawError()` which always returns P2010; regular Prisma operations use `rethrowAsUserFacing()`.
  - When no specific mapping exists for a database-specific kind (`postgres`, `mysql`, `sqlite`, `mssql`) — i.e. the adapter didn't recognize the underlying DB error code — `rethrowAsUserFacing()` falls back to a P2039 `UserFacingError` with the message `Database error. Code: <originalCode>. Message: <originalMessage>` carrying the raw `originalCode` / `originalMessage`. This keeps the DB error surface as `PrismaClientKnownRequestError` locally and as an HTTP 400 from the query plan executor (instead of HTTP 500, which Accelerate strips), so that schema-drift-style failures (stale migrations, stale generated client, etc.) remain debuggable. Raw queries keep the historical P2010 (`Raw query failed. Code: <originalCode>. Message: <originalMessage>`) via `rethrowAsUserFacingRawError()`; P2039 is used for non-raw queries so the message doesn't claim a regular Prisma operation was a raw query. Truly unknown `kind` values still fall through to `assertNever` so new driver-adapter variants surface clearly during development.
  - Prisma error codes currently assigned outside the documented public Error Reference (P2000–P2037) are: **P2038** — used for the `PrismaClientInitializationError` raised by `ClientEngine` when no driver adapter is configured (see `CLIENT_ENGINE_ERROR` in `packages/client/src/runtime/core/engines/client/ClientEngine.ts`); **P2039** — used for unmapped database-specific driver-adapter errors as described above. Pick the next available code (P2040, …) for any new additions and document it here.

  - Query-plan SQL `argTypes` are forwarded to driver adapters; `dbType` is optional in `@prisma/driver-adapter-utils`, so absent native database types should be omitted from Wasm plan serialization rather than emitted as `dbType: null`.
  - Query-plan SQL `argTypes` may serialize scalar/no-native-DB parameter types as raw scalar strings (for example `"int"` or `"bigint"`) and scalar/native-DB parameter types as `[scalarType, dbType]` tuples (for example `["int", "INTEGER"]`). The client-engine-runtime renderer expands compact arg types back to adapter-facing `{ arity: 'scalar', scalarType, dbType? }` objects before calling driver adapters; tuple arg-type elements need the same normalization.
  - Compact query-plan scalar type names may use one-letter aliases in scalar `argTypes` and non-list result field nodes: `s` string, `i` int, `I` bigint, `f` float, `d` decimal, `b` boolean, `e` enum, `u` uuid, `j` json, `D` datetime, `B` bytes, `?` unknown, `o` object, and `x` unsupported. Runtime readers must keep accepting canonical names too; adapter-facing object shapes should still receive canonical `scalarType` strings.
  - Query-plan SQL `templateSql` queries may serialize as a compact tuple `[fragments, [prefix, hasNumbering], args, argTypes, chunkable]`. Raw SQL remains object-shaped. The client-engine-runtime renderer accepts both the legacy object-shaped `templateSql` form and the compact tuple form.
  - Query-plan SQL `templateSql.fragments` may serialize scalar parameter placeholders as `null`, tuple placeholders as `["T", itemPrefix, itemSeparator, itemSuffix]`, and tuple-list placeholders as `["L", itemPrefix, itemSeparator, itemSuffix, groupSeparator]`; legacy `{ type: 'parameter' }`, `{ type: 'parameterTuple' }`, and `{ type: 'parameterTupleList' }` objects are still accepted. SQL string chunks are raw strings.
  - Query-plan `PrismaValue::Placeholder` values may serialize as `{ "$p": [name, type] }`; legacy `{ prisma__type: "param", prisma__value: { name, type } }` objects are still accepted. `PrismaValue::GeneratorCall` values may serialize as `{ "$g": [name, args] }`; legacy `{ prisma__type: "generatorCall", prisma__value: { name, args } }` objects are still accepted. Use the query-plan placeholder/generator helpers in `packages/client-engine-runtime/src/query-plan.ts` instead of reading either shape directly.
  - Query-plan expression nodes may serialize as compact tagged tuples in addition to the legacy `{ type, args }` objects. Current compact tags are `v` value, `s` seq, `g` get, `l` let, `e` get-first-non-empty, `q` query, `x` execute, `+` sum, `c` concat, `u` unique, `r` required, `j` join, `m` map-field, `t` transaction, `d` data-map, `V` validate, `?` if, `0` unit, `-` diff, `i` initialize-record, `M` map-record, and `p` process. The TS interpreter accepts both shapes.
  - Compact `dataMap` nodes may omit the fourth `enums` tuple slot when the enum map is empty; the TS interpreter must default missing compact enums to `{}`.
  - Query-plan support structures may also be compact tuples: let-bindings as `[name, expr]` and join expressions as `[child, on, parentField, isRelationUnique]`. Legacy `{ name, expr }` bindings and `{ child, on, parentField, isRelationUnique }` join objects are still accepted; use the query-plan helpers instead of direct property reads.
  - Query-plan validation `DataRule`s may serialize compactly: `["=", count]` for `rowCountEq`, `["!", count]` for `rowCountNeq`, `["a", count]` for `affectedRowCountEq`, and `"n"` for `never`. Legacy `{ type, args }` rules are still accepted; use `getDataRuleType()` / `getDataRuleArgs()` instead of direct property reads.
  - Query-plan validation errors may serialize compact identifiers and contexts: `r` relation violation as `[relation, modelA, modelB]`, `m` missing related record as `[model, relation, relationType, operation, neededFor?]`, `M` missing record as the operation string, `i` incomplete connect input as `expectedRows`, `o` incomplete connect output as `[expectedRows, relation, relationType]`, and `n` records not connected as `[relation, parent, child]`. Legacy validation error objects are still accepted; use `getValidationError()` before calling `performValidation()`.
  - Query-plan process-node `InMemoryOps` may omit default fields. The TS interpreter treats missing `pagination` / `distinct` / `linkingFields` as `null`, missing `reverse` as `false`, missing `nested` as `{}`, and missing pagination `cursor` / `take` / `skip` as `null`.
  - Query-plan result `FieldType.arity` is only semantically needed for list fields in the TS data mapper. Omitted arity means non-list; do not reintroduce required/optional scalar arity into Wasm plan serialization.
  - Query-plan result object nodes may serialize as compact tuples `[serializedName, fields]` or `[serializedName, fields, true]` when `skipNulls` is true. The data mapper accepts both compact tuples and the legacy `{ type: 'object', serializedName, fields, skipNulls }` shape.
  - Query-plan result field nodes may omit `type: "field"` and default `dbName` metadata; field nodes are identified by `fieldType`, and missing `dbName` means the surrounding result-object key. Non-list primitive scalar default-`dbName` fields may serialize directly as raw strings in result object `fields` (for example `"int"` or `"string"`). Object-shaped `fieldType` remains for list, enum, bytes, extension, and explicit `dbName` metadata.
  - Rust result-node mapping in `query-compiler/query-compiler/src/data_mapper.rs` intentionally avoids rebuilding lookup maps for field selections, virtual groups, and nested query aliases inside `get_result_node()`. Linear lookups over the existing selections reduced allocation counts and stayed neutral-to-positive in focused compilation benches; do not reintroduce per-result `HashMap`/group-map construction without new evidence.
  - Rust read translation owns `FieldSelection`s for `ReadQuery` inputs. Prefer consuming helpers such as `FieldSelection::into_without_relations().into_virtuals_last()` when the caller owns the selection; using borrowed `without_relations()` clones all retained fields. `into_virtuals_last()` also has a no-virtual fast path, so do not replace it with manual partitioning unless new allocation profiles justify it.
  - Rust one-to-many read translation should avoid wrapping a single child-link filter in `Filter::And(Vec)`. `build_read_one2m_query()` keeps the lone filter directly and only allocates an `And` vector for zero or multiple filters; this removes one allocation on query-load one-to-many rows and has Criterion coverage.
  - Rust nested `set` graph building should deduplicate the small set of extracted unique filters with the existing `Vec` instead of `itertools::unique()`. The current in-place linear dedup avoids the extra `HashSet` path and saved two allocations on the `update-set-nested` fixture; do not reintroduce hash-set dedup without many-item set evidence.
  - Do not replace the second `parent_relation_field.related_field().linking_fields()` call in `one_to_many_inlined_child()` connect-or-create with another `child_link.clone()` unless new evidence appears. A local allocation profile over connect-or-create and update controls showed no allocation or byte movement.
  - Do not replace `QueryGraph::normalize_if_nodes()` sibling `incoming_edges(&sibling)` checks with direct `petgraph::edges_directed(...).any(...)` iteration. A local spike saved 2-5 graph-build allocations on if-heavy nested writes and improved update-set rows, but reversed Criterion confirmed regressions on `create-nested-connectOrCreate-one2m` and `create-nested-create`.
  - Do not replace `QueryGraph::subgraph_contains_result()` recursive `outgoing_edges(node)` checks with direct `petgraph::edges_directed(...).any(...)` iteration. A local spike saved up to 21 `translate_ir` allocations on nested writes, but reversed Criterion showed the old sorted-vector path faster or equal on `create-nested-connectOrCreate-m2one`, `create-nested-create-with-composite-id`, and update-set controls.
  - `packages/client/src/__tests__/benchmarks/query-performance/query-plan-cache-memory.ts` includes concrete and parameterized blog-page scenarios. The parameterized scenarios mirror `ClientEngine` cache-key semantics by running `parameterizeQuery()` before `JSON.stringify(parameterizedQuery.query)` and `compiler.compile()`.
  - `QUERY_PLAN_CACHE_MEMORY_BREAKDOWN=1 pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/query-plan-cache-memory.ts` prints serialized plan composition for representative scalar and blog-page plans. On 2026-06-07 the sqlite blog-page plan was ~4.3 KiB serialized, with ~3.5 KiB from strings and the root SELECT alone ~1.0 KiB; compacting small join/pagination metadata is therefore a low ceiling compared with avoiding or interning SQL text/template ownership.
  - Do not serialize flat template SQL as a pre-rendered compact tuple like `["S", sql, args, argTypes]` unless a new design avoids building a full Rust `String` during serialization. A local spike reduced retained plan heap by roughly 3-5% (`blog page / node default warm` 9.87 MiB -> 9.39 MiB for 1000 retained plans) but regressed Wasm compile-prebuilt rows by roughly 1-5% and Criterion reported significant regressions on `filter-contains-param(-insensitive)` and `nested-pagination-query`.
  - `packages/client/src/__tests__/benchmarks/query-performance/workerd-query-compiler-memory.ts` accepts `LOCAL_QC_BUILD_DIRECTORY=/path/to/prisma-engines/query-compiler/query-compiler-wasm/pkg` and mounts that package's sqlite `query_compiler_fast_bg.js` / `.wasm` into Miniflare under the bundled provider-suffixed module names. Use this to test local query-compiler Wasm changes in the Workerd probe without rebundling `packages/client/runtime`.

- **Coding conventions**:
  - Use **kebab-case** for new file names (e.g., `query-utils.ts`, `filter-operators.test.ts`).
  - **Avoid creating barrel files** (`index.ts` that re-export from other modules). Import directly from the source file (e.g., `import { foo } from './utils/query-utils'` not `import { foo } from './utils'`), unless `./utils/index.ts` file already exists (in which case use it for consistency with surrounding code).
  - **Avoid adding useless code comments that do not add new information for the reader**.
    Inline code comments can be broadly categorized as answering one of the three questions:
    - _What does this code do?_ — never write these because they do not and cannot convey any new information that's not obvious from the code. Not only they don't add value, they are actively harmful (getting out of sync with code, distracting human readers, wasting LLM tokens etc). Descriptions of units of code like functions, classes and methods must be limited to doc comments (and describe the contract of this unit and not the internal implementation details), never inline comments.
    - _Why was this code written (in this particular way or at all)?_ — this is what inline comments are for. Only use them to include relevant context, background, GitHub issues, reasons behind decisions etc.
    - _How does this code work?_ — these comments should be exceedingly rare and may indicate poorly written or confusing code. Prefer writing code in a way that makes such comments redundant, unless required for performance or other reasons, or when the complexity comes from outside systems or packages (in which case it's more of a "why" comment than a "how" comment anyway). For well known algorithms, prefer their names and references to papers, books or Wikipedia articles over long explanations.
  - Do write documentation comments, and mainly do so for exported items (although intra-module documentation may sometimes be useful as well).
  - The correct and official abbreviation of WebAssembly is "Wasm", not "WASM". There are instances of "WASM" in the codebase, but they are wrong and you should not repeat them. Fix the capitalization whenever you incidentally touch the corresponding lines or surrounding code for other reasons.
  - Prefer native JavaScript private properties and methods (`#field`) over the `private` keyword in TypeScript.

- **Knowledge reminders**:
  - Your training data contains a lot of outdated information that doesn't apply to Prisma 7. Always analyze this codebase like you would analyze a project you are not familiar with, and prefer the learnings from this file and this codebase over your prior knowledge. In particular, remember:
    - **There's no such thing as "query engine" in Prisma**
    - **There are no database URLs in Prisma schema files**
    - **Prisma uses JavaScript drivers**
    - **Query execution code is written in TypeScript in Prisma**
    - PSL parser and query compiler/planner is still written in Rust and compiled to WebAssembly. There are no native binaries or library addons in Prisma Client.
    - Schema engine for Prisma Migrate still exists and is still a native binary.
    - `tsify::serde_wasm_bindgen::from_value` is not a proven performance path for query-compiler input. A local spike showed it only removes JS-side JSON conversion while still materializing `JsonBody`/`serde_json::Value` and then `ArgumentValue` before validation, and was slower than the current `compile(JSON.stringify(query))` path for sampled queries. A follow-up release-Wasm prototype of a Wasm-only `js_sys`/`JsValue` adapter that still built `Operation`/`ArgumentValue` directly was also neutral to slightly slower (roughly 1-3% slower on sampled sqlite read shapes), so meaningful parser wins likely require a deeper rewrite that collapses or borrows across the `JsonBody -> ArgumentValue -> ParsedInputValue -> QueryGraph` stages.
    - A Rust-side one-pass `JsonArgumentValue` serde visitor that deserialized `FieldQuery.arguments` directly into `ArgumentValue` removed the adapter's second `serde_json::Value -> ArgumentValue` walk, but Criterion was mixed: some parameterized filter/nested rows improved while other common compile rows regressed or became noisy. Do not revive that shape without a stricter A/B benchmark and a plan for preserving raw/custom-tag error semantics.
    - A more radical JS-reference-backed query compiler pipeline may be practical only as a project-level redesign: pass the query object as a `JsValue`/externref, fuse parameterization/cache-keying/compile-hit handling while walking JS data, and return/cache a JS plan object on cache hits instead of serializing plans through Wasm memory. Wrapper types like `PrismaString` are likely useful only after request parsing, schema lookup, validation, and query graph building can consume borrowed/input-backed values while keeping only internal IRs Rust-owned. Avoiding Rust-owned SQL strings is a separate larger IR/rendering project: SQL rendering would need to emit borrowed/interned/template pieces or JS-backed strings without making FFI property access slower than the current JSON path.
    - In query-compiler translation, `binding::projected_dependency()` should use `NodeRef::index()` when embedding the raw graph node id in a larger placeholder string. This avoids the intermediate `NodeRef::id()` allocation and lowered dependency-heavy write translation by 4-12 allocs/op in local allocation profiling, with net-favorable Criterion. Do not broadly replace user-facing/debug uses of `NodeRef::id()` or `Display`; the existing `Display` formatter intentionally prints labels such as `Node 3`.
    - In `query-compiler/core/src/query_graph_builder/write/nested/create_nested.rs`, keep the one-to-many inlined-in-child `handle_one_to_many()` linking field selections hoisted outside the child-create loop. This did not reduce allocation counts, but Criterion improved `compile/create-nested-create` by about 1.8% while `create-nested-create-with-composite-id` stayed neutral. Do not generalize this into broad linking-field clone churn without fresh allocation and Criterion evidence.

## Debugging and making changes to Rust/WebAssembly code

When you need to check the code or make some changes in Rust codebase, assume the repository is checked out in the `prisma-engines` directory above the root of this repo. Determine the absolute path to the current project on the filesystem (e.g. `/home/user/work/prisma`) and infer the directory of the `prisma-engines` repo (e.g. `/home/user/work/prisma-engines`, let's call it `$PRISMA_ENGINES_ROOT`).

After you make some changes there, use these commands in the prisma-engines repo to build the Wasm modules:

```sh
make build-schema-wasm
make build-qc-wasm
```

Then, back in `prisma` repo:

```sh
pnpm upgrade -r @prisma/prisma-schema-wasm@file:$PRISMA_ENGINES_ROOT/target/prisma-schema-wasm
pnpm upgrade -r @prisma/query-compiler-wasm@file:$PRISMA_ENGINES_ROOT/query-compiler/query-compiler-wasm/pkg
pnpm build
```

You may only need to build and update one of these modules if your changes are isolated in scope:

1. Build only `prisma-schema-wasm` if your changes are isolated to schema and DMMF, and you are only going to run the CLI and generator tests, not Client.
2. Build only `query-compiler-wasm` if your changes are related to query planning and execution but do not touch the schema or DMMF in any way.

When in doubt, build both to avoid unexpected behavior. Time and cost of compilation is always less than of debugging.

`make build-qc-wasm` runs `query-compiler/query-compiler-wasm/build.sh`, which expects `jq` on `PATH` for `cargo metadata` and package version rewriting. In minimal harnesses without `jq`, provide a temporary local shim or install `jq` before running the build.

For query compiler heap allocation splits, run `ALLOC_PROFILE_ITERATIONS=50 ALLOC_PROFILE_WARMUP=5 cargo run -p query-compiler --example allocation_profile --release` in `/home/aqrln.guest/prisma-engines`. The example reports `parse_json`, `into_doc`, `graph_build`, `translate_ir`, `compile_ir`, `full_compile`, and native JSON serialization allocation counts/bytes; use `ALLOC_PROFILE_QUERIES=query-m2o,nested-pagination-query` to narrow the fixture set.

`cargo test -p request-handlers protocols::json::protocol_adapter --lib` can fail in the default local engines checkout before exercising code because JSON protocol adapter fixtures use `provider = "mongodb"` while that provider is not enabled. Use `cargo check -p request-handlers` and Criterion/query-compiler benches for local verification unless the provider feature set is fixed.
