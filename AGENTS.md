# Prisma Repository – Agent Field Notes

> **Meta note**: This is the primary agent knowledge base file. `CLAUDE.md` and `GEMINI.md` are symlinks to this file—always edit `AGENTS.md` directly. When learning something new about the codebase that would help with future tasks, update this file immediately.

- **Repository scope**: Monorepo for Prisma ORM, CLI, client, tests, etc. Many packages use TypeScript, Rust and WebAssembly (via engines), TSX, and Jest; automation relies on `pnpm`. Expect large fixture directories and generated files.
- **Workspace layout**: Managed via pnpm workspaces and Turborepo (`turbo.json`). Node ^20.19 || ^22.12 || >=24.0 and pnpm >=10.15 <11 are required (see root `package.json`). Top-level scripts (`pnpm build`, `pnpm dev`, `pnpm test`) delegate into `scripts/ci/publish.ts`; package-specific commands run with `pnpm --filter @prisma/<pkg> <script>`. Turborepo caches builds; run `pnpm build` from root to build all packages in dependency order.
- **Key packages**: `packages/cli` (Prisma CLI entry point), `packages/migrate` (migrate/db namespace + fixtures), `packages/client` (client runtime), `packages/client-generator-ts` (new `prisma-client` generator), `packages/client-generator-js` (traditional `prisma-client-js` generator), `packages/client-generator-registry` (generator registry for managing generators), `packages/client-engine-runtime` (the core part of the new Rust binary free client based on Wasm query compiler, used by `ClientEngine` class in `packages/client`), `packages/client-common` (shared client utilities), `packages/client-runtime-utils` (utility types and singletons for Prisma Client), `packages/config` (`PrismaConfigInternal` + loader), `packages/internals` (shared CLI + engine glue), `packages/engines` (Rust binaries download wrapper), `packages/integration-tests` (matrix suites), `packages/query-plan-executor` (standalone query plan executor service for Prisma Accelerate), `packages/credentials-store` (credential storage utilities), sqlcommenter packages under `packages/sqlcommenter*`, and numerous driver adapters under `packages/adapter-*`.
- **Driver adapters & runtimes**: `packages/bundled-js-drivers` plus the `adapter-*` packages ship JS driver adapters: `adapter-pg`, `adapter-neon`, `adapter-libsql`, `adapter-planetscale`, `adapter-d1`, `adapter-better-sqlite3`, `adapter-mssql`, `adapter-mariadb`, `adapter-ppg` (Prisma Postgres Serverless). These are built on helpers in `driver-adapter-utils`; migrate/client fixtures exercise them, so adapter changes typically require fixture/test updates. CI tests driver adapters with flavors: `js_pg`, `js_neon`, `js_libsql`, `js_planetscale`, `js_d1`, `js_better_sqlite3`, `js_mssql`, `js_mariadb`, `js_pg_cockroachdb`.
- **Build & tooling**: Typescript-first repo with WASM/Rust assets (downloaded by `@prisma/engines`). Multiple `tsconfig.*` drive bundle vs runtime builds. Lint via `pnpm lint`, format via `pnpm format`. Maintenance scripts live in `scripts/` (e.g. `bump-engines.ts`, `bench.ts`, `ci/publish.ts` orchestrates build/test/publish flows). Build configuration uses esbuild via `helpers/compile/build.ts` with configs in `helpers/compile/configs.ts`. Most packages use `bundledConfig` which bundles to both CJS and ESM with type declarations.
- **Benchmarking**: Comprehensive performance benchmarks using Benchmark.js with CodSpeed integration. See `docs/benchmarking.md` for full documentation. Key commands:
  - `pnpm bench` - Run all benchmarks (outputs to `output.txt`)
  - `pnpm bench <pattern>` - Run benchmarks matching pattern
  - Benchmark locations:
    - End-to-end query performance: `packages/client/src/__tests__/benchmarks/query-performance/query-performance.bench.ts`
    - Query compilation/cache behavior: `packages/client/src/__tests__/benchmarks/query-performance/caching.bench.ts`
    - Query interpreter/data mapper: `packages/client-engine-runtime/bench/interpreter.bench.ts`
    - Query plan cache memory probe: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/query-plan-cache-memory.ts`
    - Local `ClientEngine` cache-hit timing probe with empty SQLite adapter: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/client-engine-cache-timing.ts`
    - Miniflare/workerd query compiler memory probe: `pnpm exec node --expose-gc --import tsx packages/client/src/__tests__/benchmarks/query-performance/workerd-query-compiler-memory.ts`
    - Client generation: `packages/client/src/__tests__/benchmarks/huge-schema/`, `packages/client/src/__tests__/benchmarks/lots-of-relations/`
  - Benchmarks run automatically on CI via `.github/workflows/benchmark.yml`; CodSpeed tracks performance over time and alerts on >100% regression.
  - The ongoing Prisma Client performance investigation journal is `wip/client-performance-journal.md`; update it after each accepted change, rejected experiment, measurement, and new lead so work survives context compactions and harness restarts.
  - For the ongoing performance work, ambitious architecture spikes are acceptable. Isolate risky or large experiments in side worktrees, use subagents for scouting/review/prototyping when that can reduce serial exploration time, and commit durable accepted/rejected outcomes back to the journal.
  - `client-engine-cache-timing.ts` supports `CLIENT_ENGINE_CACHE_TIMING_FILTER=<substring>` to run only matching rows and `CLIENT_ENGINE_CACHE_TIMING_ITERATIONS=<n>` to override row iterations. This is useful for CPU profiling one nested cached-plan row, e.g. `CLIENT_ENGINE_CACHE_TIMING_FILTER='direct plan after phase warmup' CLIENT_ENGINE_CACHE_TIMING_ITERATIONS=50000 node --cpu-prof --import tsx packages/client/src/__tests__/benchmarks/query-performance/client-engine-cache-timing.ts`.
  - `client-engine-cache-timing.ts` includes generated-client rows for public `PrismaClient` API warmed-cache calls. Use `errorFormat: 'minimal'` for these Node source rows when comparing to edge/Workerd behavior; default Node callsite capture can dominate simple generated-client timings and hide real proxy/runtime changes.
  - For Cloudflare/workerd probes, instantiate query compiler Wasm through a static module import / Miniflare `CompiledWasm` module. Dynamic `new WebAssembly.Module(bytes)` inside the worker fails with `Wasm code generation disallowed by embedder`. When importing `packages/client/runtime/wasm-compiler-edge.mjs` in Miniflare, explicitly register the built `@prisma/client-runtime-utils` ESM under that bare module specifier. Keep manual compiler and generated-client rows in separate Miniflare instances because both instantiate the wasm-bindgen query compiler runtime. Workerd's worker-internal timer can be too coarse for warmed pure-JS request loops; use host `dispatchFetch` timing as an upper-bound signal.
  - In `workerd-query-compiler-memory.ts`, generated-client precomputed counters must wrap both `requestWithPrecomputedQueryPlanCacheHit()` and `requestPrecomputedCachedResult()`. Direct engine-precomputed rows use the latter on descriptor hits, so omitting it makes active engine fast-path runs print `precomputed fast path: hits 0`.
  - For query-compiler graph traversal micro-optimizations, require Criterion evidence in addition to allocation counts. Direct `petgraph` iteration and helper-level `SmallVec` experiments around `subgraph_contains_result()`, `is_direct_child()`, and `direct_child_pairs()` either had no allocation effect or regressed/failed to improve nested-create benches despite saving one or two allocations.
  - For query-compiler translation allocation work, removing expression scaffolding that `Expression::simplify()` would discard can be worthwhile only with timing proof: the accepted single-result-subgraph fast path in `fold_result_scopes()` saved four nested-create allocations and improved the focused rerun. In contrast, rewriting `process_children()` result-position splitting saved one to six allocations on nested writes but repeatedly regressed Criterion rows; both a reverse-index removal patch and a `None` / `One` / `Many` result-subgraph enum variant were rejected. Returning binding-only child `Let` nodes directly from `process_child_with_dependencies()` saved two allocations but also regressed nested-create Criterion.
  - Do not retry the 2026-06-08 expression-container sidecar shapes without a new CPU hypothesis: replacing `Cow<'static, str>` expression binding names with a typed `BindingName` enum saved nested-write allocation counts but regressed Criterion (`update-set-nested` +7.25%), and skipping discarded result `Let` nodes in `wrap_children_with_expr()` had tiny allocation wins with broad timing regressions.
  - For nested cached-plan runtime work, skipping `klona()` for static `InMemoryOps` in compact `p` / legacy `process` nodes was rejected: scan-only and WeakMap-cached static/dynamic guards did not clearly improve cached wrapper, inner-plan, or phase-warmed direct-plan nested rows.
  - For `QueryInterpreter` query leaves, guarded one-rendered-query fast paths are viable: legacy `query` / `execute` and compact `q` / `x` can bypass the generic rendered-query loop when SQL comments and query instrumentation are inactive. This improved/kept healthy the nested-row direct/local/inner timing rows and the standalone interpreter bench.
  - For compact `QueryInterpreter` nodes, the per-interpreter compiled-node WeakMap is an accepted cached-plan win. It compiles hot compact tags (`v`, `s`, `g`, `l`, `e`, `q`, `x`, `u`, `m`, `j`, `d`, `p`) into closures, preserves the direct `d(q)` / `d(u(q))` data-map fast paths, and falls back to the old compact interpreter for unsupported tags.
  - `QueryPlanCache` keeps a last-hit entry for single and batch lookups to avoid repeated `Map.get()` calls on warmed same-shape cache hits. Preserve the eviction invalidation tests if touching this path, because stale last-hit entries can otherwise bypass the LRU maps after eviction.
- **Testing & databases**: `TESTING.md` covers Jest/Vitest usage. Most suites run as `pnpm --filter @prisma/<pkg> test <pattern>`. DB-backed tests expect `.db.env` and Docker services from `docker/docker-compose.yml` (`docker compose up -d`). Client functional tests sit in `packages/client/tests/functional`—run them via `pnpm --filter @prisma/client test:functional` (with typechecking) or `pnpm --filter @prisma/client test:functional:code` (code only); `helpers/functional-test/run-tests.ts` documents CLI flags to target providers, drivers, etc. Client e2e suites require a fresh `pnpm build` at repo root, then `pnpm --filter @prisma/client test:e2e --verbose --runInBand`. The legacy `pnpm --filter @prisma/client test` command primarily runs the older Jest unit tests plus `tsd` type checks. Migrate CLI suites live in `packages/migrate/src/__tests__`, the CLI runs both Jest (legacy suites) and Vitest (new subcommand coverage) via `pnpm --filter prisma test`, and end-to-end coverage lives in `packages/integration-tests`.

- **Client performance spike notes**:
  - A 2026-06-08 `client-engine-cache-timing.ts` spike in `/home/aqrln.guest/prisma-js-owned-hit-spike` found that generated descriptor cache-hit prototypes need direct JS property access or generated shape guarantees. Generic/exact `Object.keys()` descriptor walks are too expensive for the hot path; a sentinel/direct-access blog-page descriptor measured about 1 us/op for extraction on Node, and a direct cached-plan descriptor matched the static-shape executable lower bound. The prototype is not production-safe as written because it does not prove absence of extra enumerable keys.
  - A 2026-06-08 Workerd spike in `/home/aqrln.guest/prisma-workerd-plan-descriptor-spike` confirmed the same direction on the target runtime: sentinel/direct descriptor extraction measured about 0.40 us/op worker-loop for the nested blog-page shape versus about 0.90 static exact-shape and 1.60 lazy interpreted descriptor extraction. A product-shaped sentinel path through `requestPrecomputedCachedResult()` measured about 7.40 us/op versus about 8.70 for engine-precomputed static protocol and about 13.00 for the default generated blog-page row. Treat this as benchmark evidence only; product code still needs exact-shape safety, fallback semantics, and batching-aware routing.

- **Client architecture (Prisma 7)**:
  - `ClientEngine` in `packages/client/src/runtime/core/engines/client/` orchestrates query execution using Wasm query compiler.
  - `ClientEngine` query plan cache defaults are build-target specific: Node.js `client` builds keep up to 1000 entries, while `wasm-compiler-edge` builds keep up to 100 entries by default to reduce retained memory. `queryPlanCacheMaxSize: 0` disables the cache.
  - `ClientEngine.request()` and `requestBatch()` use a synchronous connected-engine fast path before falling back to async `#ensureStarted()`; keep that path when editing hot request flow because it avoids an avoidable async hop on warmed clients.
  - Keep disabled debug logging out of `ClientEngine` warmed request hot paths. `@prisma/debug` records call arguments in a global history even when the namespace is not enabled, so unguarded per-request `debug(...)` calls can retain query plans and add microseconds of overhead.
  - `applyModel()` intentionally wraps `modelActionsLayer(client, dmmfModelName)` in `cacheProperties()`. Do not remove this without generated-client A/B evidence: a same-session 2026-06-07 run moved generated-client `findUnique` from about 22.62 to 17.85 us/op and nested blog-page from about 52.71 to 47.46 us/op while preserving fresh per-call `PrismaPromise` behavior.
  - `applyFluent()` intentionally hoists relation keys/proxy own-key metadata and reuses the root empty data path; `modelActionsLayer()` intentionally reuses fluent-supplied callsites; `getCallSite('minimal')` and edge disabled-callsite paths intentionally return a singleton. A 2026-06-07 generated-client A/B moved `findUnique` about 17.65 -> 16.85 us/op and nested blog-page about 47.85 -> 46.74 us/op, with a Workerd high-iteration smoke at 17.49/39.84 us/op host upper bounds after rebuild.
  - `applyFluent()` also intentionally precomputes reusable non-`get` proxy traps instead of calling `defaultProxyHandlers(proxyOwnKeys)` for every returned PrismaPromise. A 2026-06-07 A/B moved simple generated-client `findUnique` about 15.73 -> 14.37/14.12 us/op; nested blog-page stayed noisy/mixed, so re-benchmark both rows if touching this shape.
  - Do not add a fluent proxy `get` fast path for `then` / `catch` / `finally` / `requestTransaction` / `spec` without new evidence. A 2026-06-07 spike was mixed: patched exact generated rows were 4.83 / 17.62 us/op, while adjacent reverted control was 4.94 / 17.43 us/op, so the added branch did not clearly improve product timing.
  - `createPrismaPromiseFactory()` intentionally returns a `PrismaPromiseImpl` class with prototype methods instead of an object literal with per-instance `then`/`catch`/`finally`/`requestTransaction` closures. A 2026-06-07 A/B moved generated-client `findUnique` about 14.10 -> 12.62/12.45 us/op and nested blog-page about 39.14 -> 36.75/36.61 us/op; Workerd generated-client smoke after rebuild was 15.24/32.67 us/op host upper bounds.
  - Do not inline the no-transaction branch inside `PrismaPromiseImpl.then()` or drop its `onFulfilled ?? undefined` / `onRejected ?? undefined` normalization without new evidence. Local spikes were noise or mixed: no-transaction inline measured about 5.88/22.10 us/op vs a 5.89/22.12 baseline, and direct handler passthrough measured about 5.94/21.93 with a simple-row regression.
  - `getPrismaClient()._request()` intentionally has an empty-extension fast path that still creates the operation span and Node `AsyncResource`, then calls `_executeRequest()` directly. This skips middleware-shaped params, query-extension dispatch, result-extension context resolution, and `applyAllResultExtensions()` when `this._extensions.isEmpty()`. A 2026-06-07 generated-client A/B moved `findUnique` about 12.38 -> 10.97 us/op and nested blog-page about 37.11 -> 35.62 us/op; Workerd smoke after rebuild was 14.91/32.77 us/op host upper bounds.
  - `DataLoader` intentionally schedules batch dispatch with `queueMicrotask()` instead of `process.nextTick()`. It still batches same-turn requests and splits requests separated by `await`, with coverage in `DataLoader.test.ts`. A 2026-06-07 generated-client A/B moved `findUnique` about 10.99 -> 10.57 us/op and nested blog-page about 35.57 -> 34.13 us/op; Workerd after rebuild moved generated-client host upper bounds to 12.31/32.34 us/op and worker-internal timings to 10.41/29.75 us/op.
  - `DataLoader` supports an optional cheap `canBatch` predicate so `RequestHandler` can defer expensive structural `getBatchId()` calls for transactions and `findUnique` / `findUniqueOrThrow` requests. Single batchable candidates still wait for the microtask boundary but go to `singleLoader` without computing the recursive sorted selection key; multiple same-turn candidates compute `batchBy` at dispatch and batch normally. A 2026-06-07 A/B moved generated-client `findUnique` about 9.10 -> 6.77 us/op and nested blog-page about 32.63 -> 24.18 us/op; Workerd worker-internal timings after rebuild were 6.85/21.90 us/op.
  - `getPrismaClient()._executeRequest()` intentionally checks disabled debug logging through `isClientDebugEnabled()`, mirroring `ClientEngine`'s cheap global-debug guard before calling `Debug.enabled('prisma:client')`. A 2026-06-07 generated-client A/B moved `findUnique` about 10.36 -> 9.30 us/op and nested blog-page about 34.47 -> 33.55 us/op; Workerd after rebuild moved host upper bounds to 11.26/29.53 us/op and worker-internal timings to 9.39/26.85 us/op.
  - Do not inline `DynamicTracingHelper`'s disabled fallback with optional chaining against `getGlobalTracingHelper()` without new evidence. A local spike replacing `this.getTracingHelper().runInChildSpan(...)` with `getGlobalTracingHelper()?.runInChildSpan(...) ?? callback()` regressed generated-client rows to about 6.46/24.51 us/op.
  - Do not replace `DataLoader.dispatchBatches()` `.then(...).catch(...)` continuations with two-argument `.then(success, failure)` without new evidence. A post-build source run regressed generated-client rows to about 9.61/33.57 us/op, and same-session A/B was mixed (`findUnique` 9.42 -> 9.27 us/op, nested blog-page 32.68 -> 32.78 us/op).
  - `getBatchId()` intentionally uses a loop-based sorted-key builder instead of `sort().map().join()` to keep findUnique batching keys cheaper on generated-client hot paths. A DataLoader continuation/lookup cleanup tried at the same time was reverted because it weakened the product-path timing.
  - Do not replace `getBatchId()` recursive template literals with `+` concatenation without new evidence. A local A/B regressed generated-client rows from about 9.27/32.58 us/op to 9.39/33.15 us/op.
  - Do not add a `keysArray.length > 1` guard before `getBatchId()` sorting without new evidence. A local spike initially looked weakly positive, but post-build same-session source A/B regressed generated-client rows from about 9.45/33.47 us/op to 9.63/34.38 us/op.
  - `serializeJsonQuery()` intentionally creates nested `SerializeContext`s lazily for explicit `select` / `include` traversal. Scalar `true`/`false` selections should not allocate nested contexts or path arrays unless strict-undefined validation or relation/object recursion needs them. A 2026-06-07 A/B moved generated-client `findUnique` about 16.65 -> 15.58 us/op and nested blog-page about 45.79 -> 40.60 us/op, with Workerd smoke at 16.96/35.24 us/op host upper bounds after rebuild.
  - `serializeJsonQuery()` also fast-paths successful primitive argument values in objects and arrays (`null`, string, number, boolean, bigint) before creating nested argument contexts. Keep strict-undefined and invalid-value paths context-backed. A 2026-06-07 A/B was noisy on simple `findUnique` but moved nested blog-page about 41.26 -> 40.84 us/op, with Workerd smoke at 16.43/34.96 us/op host upper bounds after rebuild.
  - `serializeJsonQuery()` should not re-check `isSkip(value)` in `createExplicitSelection()` after the first `isSkip()` branch has already continued. A 2026-06-07 generated-client A/B moved `findUnique` about 9.54 -> 9.44 us/op and nested blog-page about 33.30 -> 32.73 us/op; Workerd stayed in the same band at 11.47/29.93 us/op host upper bounds.
  - `serializeJsonQuery()` selection traversal intentionally uses `Object.keys()` indexed loops instead of `Object.entries()` in include/omit/explicit-select paths. This preserves own enumerable string-key semantics while avoiding per-entry pair arrays. A post-lazy-batch-key 2026-06-07 Node A/B moved generated-client rows from about 7.01/24.75 us/op to 6.72/23.39 us/op; Workerd was neutral rather than clearly positive.
  - `isSkip(value)` intentionally compares by `value === skip` rather than `value instanceof Skip`. `Skip` construction is guarded by a module-private symbol and supported values come from the exported singleton / `skip.ifUndefined()`, so identity is the intended fast path. A 2026-06-07 generated-client A/B was modest but net-positive after reapply (`findUnique` about 6.47 -> 6.38 us/op, nested blog-page about 23.48 -> 23.15 us/op), and Workerd worker-internal timings were 6.57/19.85 us/op after rebuild.
  - `SerializeContext.isRawAction()` intentionally uses a switch instead of constructing an action array and calling `.includes()` on every implicit selection. A 2026-06-07 same-session A/B moved generated-client rows from about 5.94/22.31 us/op to 5.84/22.16 us/op; Workerd stayed in band at 6.69/19.80 us/op worker-internal.
  - Do not guard `computeEngineSideSelection(select, computedFields)` with `computedFields ? ... : select` in `createExplicitSelection()` without new evidence. A local A/B was neutral-to-negative on nested generated-client rows (baseline about 9.36/32.31 us/op, patched about 9.30/32.48 us/op).
  - `SerializeContext.findField()` intentionally uses a WeakMap-cached field-name table per runtime model and stores the resolved table on each context, instead of scanning `modelOrType.fields` or hitting the WeakMap for every lookup. Earlier field-map attempts were rejected before the nested serializer row existed, but 2026-06-07 retries with isolated serializer evidence moved nested serialization from 4.24 to 3.54-3.55 us/op with the map, then to 3.38 us/op with context-local field tables; Workerd stayed in band, with the context-local follow-up at 6.16 / 17.85 us/op on the confirming run.
  - Do not thread a `hasExtensions` boolean through `SerializeContext` just to skip `extensions.getAllComputedFields()` on empty-extension clients without new evidence. A 2026-06-07 same-session A/B was neutral-to-negative: patched generated-client rows were 6.13/22.33 us/op versus reverted 6.05/22.28 us/op.
  - Do not pass already-looked-up `RuntimeField` values into `SerializeContext.nestSelection()` without new evidence. A local split spike improved nested generated-client rows but made the simple generated `findUnique` row noisy/worse (`6.12` us/op field-reuse-only vs `5.94` baseline), while a narrower `isRawAction()` switch kept the broad signal cleaner.
  - `QueryPlanCache` interns repeated string values only for join/raw-nested-shaped plans and only for strings with length >= 128. The interner is refcounted per cache entry so updates, evictions, and `clear()` release strings from old plans; retained entries store compact flat refcount arrays instead of `Map`s. Avoid broad all-plan string interning because scalar-selection plans mostly have unique SQL strings and can regress from bookkeeping overhead.
  - Do not split cached `templateSql.fragments` strings inside `QueryPlanCache` just to expose common SQL suffixes or selected-column chunks for interning. A local spike splitting long strings at `FROM` and then at comma boundaries passed cache tests but did not reduce retained V8 heap for blog-page plans and worsened larger 1,000-shape rows.
  - `client-engine-cache-timing.ts` has cached request wrapper rows using real `QueryPlanCache` + `LocalExecutor`, plus a scripted nested blog-page adapter that returns one post, author/category, two tags, and two comments/comment authors (seven SQL calls per request). After the disabled-debug guard, generic `ClientEngine` wrapper glue is a small part of warmed cache-hit cost; bigger remaining targets are cache-key/parameterization and cached-plan execution/rendering.
  - `client-engine-cache-timing.ts` repeats the full `ClientEngine` blog-page nested warmed-cache row after the phase-split rows. Prefer this late row when judging full-engine overhead: local runs showed the early warmed row was order/JIT sensitive (about 68-76 us/op early vs about 52-57 us/op late), while the late row stayed about 8 us/op above cached request wrapper rows.
  - `client-engine-cache-timing.ts` has `generated client serialize ...` rows that construct generated-style args and call `serializeJsonQuery()` without DataLoader or engine execution. A 2026-06-07 run measured about 0.32 us/op for simple `findUnique` and 4.17 us/op for the nested blog-page shape, versus full generated rows at 4.94 / 18.24 us/op. Nested serializer work is still a meaningful target; simple scalar serialization is not.
  - `client-engine-cache-timing.ts` also has `generated client serialize cache key ...` rows that time generated-style args through `serializeJsonQuery()`, `parameterizeQuery()`, native `JSON.stringify()`, and single-query cache-key construction. A 2026-06-07 run measured 1.71 us/op for `findUnique` and 7.19 us/op for the nested blog-page shape, while the adjacent full generated nested row was 18.58 us/op. This is the measured ceiling for replacing the generated-client front half with a static descriptor or fused serializer/parameterizer.
  - `client-engine-cache-timing.ts` has benchmark-only `generated client static descriptor extract ...` and `cached request wrapper static descriptor ...` rows for the nested blog-page generated args shape. A guarded exact-shape descriptor that validates `where/select`, static nested selections, `comments.take`, and `orderBy` measured about 1.16 us/op to extract `%1`; executing the cached wrapper with that descriptor measured about 8.02-8.06 us/op versus the static-shape lower bound around 7.05 us/op. Static descriptors are a plausible cache-hit architecture lead if they remain guarded and fall back to `serializeJsonQuery()` on any semantic uncertainty.
  - A 2026-06-08 side-worktree spike in `/home/aqrln.guest/prisma-generated-shape-probe-spike` (commit `6dbdc62fe`) added an optional generated-query-shape probe hook and manually injected a safe exact blog-page probe in `client-engine-cache-timing.ts`. It improved the generated engine-precomputed blog-page row from 11.52 to 11.24 us/op and the request-precomputed blog-page row from 12.94 to 12.01 us/op over 100k Node iterations, while preserving batched `findUnique` behavior (`queryRaw=100000`, `precomputedBatchHits=200000`). Treat this as a positive lead, not landed product design: the unresolved work is generator emission / shape-exactness for arbitrary user args, not the request/engine execution path.
  - Do not land the generated-query-shape probe hook as a per-model/action config map. A 2026-06-08 generator/runtime review found it unsafe for productization: generators do not know runtime callsite object shapes, and a per-action probe can be applied to the wrong learned descriptor. A viable version needs descriptor-bound exact matchers tied to the specific learned protocol query and placeholder mapping, with recursive own-key exactness checks and Worker-safe static code only; avoid `eval` / `new Function` and combinatorial generated probe emission.
  - Descriptor-bound exact matchers remain a positive generated-cache-hit lead, but only with descriptor identity as the safety boundary. - **Driver adapter error handling**:

  - `rethrowAsUserFacing()` in `packages/client-engine-runtime/src/user-facing-error.ts` maps `MappedError` kinds to Prisma error codes (P2xxx).
  - To add a new error mapping: (1) add kind to `MappedError` in driver-adapter-utils, (2) map database error code in relevant adapter(s), (3) add Prisma code mapping in `getErrorCode()` and message in `renderErrorMessage()`.

  - When no specific mapping exists for a database-specific kind (`postgres`, `mysql`, `sqlite`, `mssql`) — i.e. the adapter didn't recognize the underlying DB error code — `rethrowAsUserFacing()` falls back to a P2039 `UserFacingError` with the message `Database error. Code: <originalCode>. Message: <originalMessage>` carrying the raw `originalCode` / `originalMessage`. This keeps the DB error surface as `PrismaClientKnownRequestError` locally and as an HTTP 400 from the query plan executor (instead of HTTP 500, which Accelerate strips), so that schema-drift-style failures (stale migrations, stale generated client, etc.) remain debuggable. Raw queries keep the historical P2010 (`Raw query failed. Code: <originalCode>. Message: <originalMessage>`) via `rethrowAsUserFacingRawError()`; P2039 is used for non-raw queries so the message doesn't claim a regular Prisma operation was a raw query. Truly unknown `kind` values still fall through to `assertNever` so new driver-adapter variants surface clearly during development.
  - Prisma error codes currently assigned outside the documented public Error Reference (P2000–P2037) are: **P2038** — used for the `PrismaClientInitializationError` raised by `ClientEngine` when no driver adapter is configured (see `CLIENT_ENGINE_ERROR` in `packages/client/src/runtime/core/engines/client/ClientEngine.ts`); **P2039** — used for unmapped database-specific driver-adapter errors as described above. Pick the next available code (P2040, …) for any new additions and document it here.

- **Coding conventions**:
  - Use **kebab-case** for new file names (e.g., `query-utils.ts`, `filter-operators.test.ts`).
  - **Avoid creating barrel files** (`index.ts` that re-export from other modules). Import directly from the source file (e.g., `import { foo } from './utils/query-utils'` not `import { foo } from './utils'`), unless `./utils/index.ts` file already exists (in which case use it for consistency with surrounding code).
  - **Avoid adding useless code comments that do not add new information for the reader**.
    Inline code comments can be broadly categorized as answering one of the three questions:
    - _What does this code do?_ — never write these because they do not and cannot convey any new information that's not obvious from the code. Not only they don't add value, they are actively harmful (getting out of sync with code, distracting human readers, wasting LLM tokens etc). Descriptions of units of code like functions, classes and methods must be limited to doc comments (and describe the contract of this unit and not the internal implementation details), never inline comments.
    - _Why was this code written (in this particular way or at all)?_ — this is what inline comments are for. Only use them to include relevant context, background, GitHub issues, reasons behind decisions etc.
    - _How does this code work?_ — these comments should be exceedingly rare and may indicate poorly written or confusing code. Prefer writing code in a way that makes such comments redundant, unless required for performance or other reasons, or when the complexity comes from outside systems or packages (in which case it's more of a "why" comment than a "how" comment anyway). For well known algorithms, prefer their names and references to papers, books or Wikipedia articles over long explanations.
  - Do write documentation comments, and mainly do so for exported items (although intra-module documentation may sometimes be useful as well).
  - The correct and official abbreviation of WebAssembly is "Wasm", not "WASM". There are instances of "WASM" in the codebase, but they are wrong and you should not repeat them. Fix the capitalization whenever you incidentally touch the corresponding lines or surrounding code for other reasons.
  - Prefer native JavaScript private properties and methods (`#field`) over the `private` keyword in TypeScript.

A 2026-06-08 numeric-only tuple-returning spike passed focused interpreter tests but regressed exact compact blog-page rows to 6.74 us/op versus the accepted wrapper-fast-path row's 6.25 us/op. The next raw-nested project should be a real flat result-set assembly program or a different hot path, not another wrapper around the current tree.
    - Do not change `applyModel.ts` lazy descriptor object nodes from `fields: Record<string, LazyDescriptorNode>` to key-aligned field arrays without new evidence. A 2026-06-08 spike regressed generated blog-page precomputed fast paths to 13.00 us/op for engine-precomputed and 14.66 us/op for request-precomputed, so the current record shape is better on V8.

## Debugging and making changes to Rust/WebAssembly code

When you need to check the code or make some changes in Rust codebase, assume the repository is checked out in the `prisma-engines` directory above the root of this repo. Determine the absolute path to the current project on the filesystem (e.g. `/home/user/work/prisma`) and infer the directory of the `prisma-engines` repo (e.g. `/home/user/work/prisma-engines`, let's call it `$PRISMA_ENGINES_ROOT`).

After you make some changes there, use these commands in the prisma-engines repo to build the Wasm modules:

```sh
make build-schema-wasm
make build-qc-wasm
```

Then, back in `prisma` repo:

```sh
pnpm upgrade -r @prisma/prisma-schema-wasm@file:$PRISMA_ENGINES_ROOT/target/prisma-schema-wasm
pnpm upgrade -r @prisma/query-compiler-wasm@file:$PRISMA_ENGINES_ROOT/query-compiler/query-compiler-wasm/pkg
pnpm build
```

You may only need to build and update one of these modules if your changes are isolated in scope:

1. Build only `prisma-schema-wasm` if your changes are isolated to schema and DMMF, and you are only going to run the CLI and generator tests, not Client.
2. Build only `query-compiler-wasm` if your changes are related to query planning and execution but do not touch the schema or DMMF in any way.

When in doubt, build both to avoid unexpected behavior. Time and cost of compilation is always less than of debugging.

`make build-qc-wasm` runs `query-compiler/query-compiler-wasm/build.sh`, which expects `jq` on `PATH` for `cargo metadata` and package version rewriting. In minimal harnesses without `jq`, provide a temporary local shim or install `jq` before running the build.

For query compiler heap allocation splits, run `ALLOC_PROFILE_ITERATIONS=50 ALLOC_PROFILE_WARMUP=5 cargo run -p query-compiler --example allocation_profile --release` in `/home/aqrln.guest/prisma-engines`. The example reports `parse_json`, `into_doc`, `graph_build`, `translate_ir`, `compile_ir`, `full_compile`, and native JSON serialization allocation counts/bytes; use `ALLOC_PROFILE_QUERIES=query-m2o,nested-pagination-query` to narrow the fixture set.

`cargo test -p request-handlers protocols::json::protocol_adapter --lib` can fail in the default local engines checkout before exercising code because JSON protocol adapter fixtures use `provider = "mongodb"` while that provider is not enabled. Use `cargo check -p request-handlers` and Criterion/query-compiler benches for local verification unless the provider feature set is fixed.
